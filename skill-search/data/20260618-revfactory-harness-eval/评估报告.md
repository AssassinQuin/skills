## 核心优点

### 1. 方法论严谨，Phase 0-7 闭环
- **Phase 0 现状审计**：触发即读 `.claude/agents/` `.claude/skills/` `CLAUDE.md`，分三路（新构筑/扩展/运维）
- **Phase 2 双维度决策**：执行模式（Team/Sub/Hybrid）× 架构模式（Pipeline/Fan-out/Expert Pool/Producer-Reviewer/Supervisor/Hierarchical）
- **Phase 3-0 / 4-0 中复检**：新生成前强制对比已有 agent/skill，防 drift（这点比 skill-creator 更系统）
- **Phase 7 持续演化**：用户反馈分类映射到修改对象（结果/角色/流程/团队/触发器）+ CLAUDE.md 变更历史表

### 2. QA Agent 模式独到
- 强制 `general-purpose` 类型（不用 `Explore`，因 read-only 跑不了验证脚本）
- 核心是**"边界交叉对比"**（API response vs frontend hook shape），不是"存在确认"
- **incremental QA**：模块完成即查，不是全栈完成一次查
- 基于真实项目 7 个 bug 案例，参考价值高

### 3. 工厂-生态位划分清晰
README 的 "Coexistence" 表区分了：
- **Archon**（同层、邻居子层：Runtime-Configuration Factory）
- **meta-harness**（同概念、Codex runtime 移植）
- **wshobson/agents**（零件目录，harness 用它当素材）
- **LangGraph**（不同轨道）

这种自我定位比大多数 skill 更诚实，避免"什么都做"的过度宣称。

### 4. 触发器"pushy description"原则
明确反对被动 description，给出对比示例：
- 坏：`"PDF 文档处理 skill"`
- 好：`"PDF 读取/提取/合并/分割/旋转/水印/加密/OCR 全覆盖，提到 .pdf 文件或要求 PDF 产出时必须使用"`

与 skill-creator 的 description 指南一致但更激进。

## 核心缺点（与用户环境的冲突）

### ⚠️ 1. 强制 `model: "opus"` — **直接违反用户 coding-rules R5.1**
SKILL.md 明确写：
> 모든 에이전트는 `model: "opus"`를 사용한다. ... 하네스의 품질은 에이전트의 추론 능력에 직결되며, opus가 최고 품질을 보장한다.

**与用户规则的冲突**：
- 用户 coding-rules R5.1：`不需要推理→haiku；需要推理但非战略性→sonnet；战略性→opus`
- harness 的强制 opus 会让 QA、日志解析等机械任务全用 opus，**烧钱 + 慢**
- 已有 Issue [#29](https://github.com/revfactory/harness/issues/29) 和 [#18](https://github.com/revfactory/harness/issues/18) 都在抗议这点，作者尚未接受

### ⚠️ 2. SKILL.md 主体是韩文
- frontmatter `description` 是韩文（`하네스를 구성합니다...`）
- 正文小标题、表格、checklist 部分韩英混排
- 英文用户触发器命中率会显著下降（Claude 看韩文 description 不会优先匹配英文 prompt）

### ⚠️ 3. 依赖实验特性 Agent Teams
- 要求 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
- 团队模式（TeamCreate/SendMessage/TaskCreate）尚未稳定
- Issue #20 反映 marketplace 安装路径有 bug

### ⚠️ 4. 强制生成 agent 定义文件
SKILL.md 写：
> 빌트인 타입(general-purpose, Explore, Plan)을 사용하더라도 에이전트 정의 파일은 생성한다.

**与用户当前架构冲突**：
- 用户 `~/.claude/agents/` 是**全局 subagent 体系**（brainstorm-collider / evolver-auditor / oracle / researcher / reviewer 等），与项目解耦
- harness 强制在**每个项目的** `.claude/agents/` 创建定义，会与用户已有的全局体系产生重复

### ⚠️ 5. A/B 数据可信度
- "+60% avg quality, 15/15 win-rate, -32% variance" 来自**作者自测**
- README 自己也承认："n=15, author-measured, third-party replications pending"
- 作者 Hwang, M. 引用了自己 2026 论文，但论文无同行评审

### ⚠️ 6. 中文社区零讨论
中文 "harness" 一词已被 OpenAI 的 **Harness Engineering** 范式（2026-02 提出）占据，知乎/掘金/V2EX/CSDN 全在讨论那个概念，对 revfactory/harness 仓库**零讨论**。这意味着无中文踩坑经验可参考。

## 安装建议

### 不建议直接纳入 `/Users/ganjie/skills/`，原因：
1. 与用户 **coding-rules R5.1** 冲突（强制 opus）
2. 与用户**全局 agent 体系**冲突（强制每项目 `.claude/agents/`）
3. SKILL.md 韩文为主，触发器命中率低
4. 用户已有覆盖此能力的 skills：
   - `skill-creator`（单 skill 创建）
   - `huashu-nuwa`（meta-skill 生成人物视角）
   - `darwin-skill` / `skill-evolver`（优化已有 skill）
   - `planning-with-files`（执行期管理）

### 建议借鉴的做法：
1. **references/agent-design-patterns.md** — 6 种架构模式 + 决策树，**强烈建议下载到本地知识库**作为方法论参考
2. **references/orchestrator-template.md** — orchestrator 设计模板（含 `_workspace/` 文件命名约定、Phase 0 上下文确认、错误处理 1-retry-then-skip）
3. **QA Agent incremental 模式** — 模块完成即查，不是全栈一次查
4. **"pushy description" 原则** — 写 description 时显式列出触发场景
5. **CLAUDE.md 只放指针 + 变更历史**，不放 agent/skill 列表（避免重复）

### 如果一定要安装（仅在需要做横向对比时）：
```bash
# 作为 plugin（不污染全局 skills 库）
/plugin marketplace add revfactory/harness
/plugin install harness@harness-marketplace

# 或克隆到隔离目录做研究
git clone https://github.com/revfactory/harness ~/research/harness-ref
```

## 适合谁

✅ **适合**：
- 想搭建完整 agent 团队（不是单 skill）的 Claude Code 重度用户
- 接受韩文 SKILL.md + 实验 Agent Teams 特性的早期采用者
- 不在乎 token 成本，全用 opus 的用户

❌ **不适合**：
- 用户当前架构（全局 subagent + 模型分层 + 中文为主）
- 需要 Codex / Gemini 跨运行时（仅 Claude Code 原生，Codex 移植在 meta-harness）
- 预算敏感场景（强制 opus 会显著放大成本）
