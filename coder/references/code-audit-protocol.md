# Code Audit Protocol — 子 agent 深度审计协议

由 coder skill 审查门控 Layer 2 触发，或用户直接触发"审计代码"意图时加载。

## 1. 审计触发条件

| 触发方式 | 判定 |
|---------|------|
| 用户意图含"审计"/"audit"/"review diff"/"审查代码变更" | 直接触发 |
| 完整路径 + 复杂度=复杂 | 自动触发 |
| 标准/快速路径 | 不触发 |

## 2. 子 agent 委托协议

### 输入（由主 agent 提供）

```json
{
  "files": ["改动文件相对路径列表"],
  "diff": "git diff 输出（仅 staged/unstaged 变更部分）",
  "language": "go|python|...",
  "complexity": "简单|标准|复杂",
  "project_context": "框架/工具链信息（如 {框架名} {版本号}）",
  "audit_scope": "full|用户指定范围"
}
```

### 前置校验（审计前必须执行）

- `files` 列表非空 → 否则报错终止
- 每个文件可读取 → 不可读的跳过并记录
- `diff` 非空 → 空则降级为自审
- 测试文件可发现（同目录 `*_test.go` / `test_*.py`）→ 不可发现标注 `[测试未保护]`

## 3. 代码类型分类审计策略

**核心原则：不同代码类型有不同的审计重点，不可一刀切。**

### A. 金额/精度代码（涉及 decimal/float/货币/金额/积分）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| A1 | 精度处理是否使用 decimal（禁止 float64 做金额运算） | 所有金额变量类型为 decimal |
| A2 | 加减乘除是否有溢出保护 | 大数运算有范围检查或使用安全库 |
| A3 | 货币/积分转换是否正确（汇率、单位换算） | 转换方向正确，无遗漏除法/乘法 |
| A4 | 零值和负值边界处理 | 金额为零时有合理分支，负值有防护 |
| A5 | 并发场景下余额更新是否原子 | 事务内更新，无竞态窗口 |

### B. 并发/goroutine 代码（goroutine/channel/mutex/context）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| B1 | goroutine 是否有泄漏风险 | 有退出机制（context cancel / done channel） |
| B2 | 共享状态是否有竞态保护 | mutex / atomic / channel 保护，无裸读写 |
| B3 | channel 是否正确关闭 | 生产者关闭，消费者不写已关闭 channel |
| B4 | context 取消传播是否完整 | 子操作尊重 ctx.Done()，不吞掉 cancel |
| B5 | deadlock 风险 | 锁获取顺序一致，无循环等待 |

### C. API 签名变更（函数签名/接口/HTTP handler）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| C1 | 向后兼容性 | 新增参数有默认值或新增重载，旧调用方不受影响 |
| C2 | 参数校验完整性 | 必填参数有 nil/空检查，类型正确 |
| C3 | 错误码/错误信息一致性 | 与同模块其他接口的错误格式一致 |
| C4 | 返回值语义清晰 | 无歧义的返回结构，调用方不需要猜含义 |

### D. 数据库操作（SQL/事务/ORM/DAO）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| D1 | 事务完整性 | 相关操作在同一事务内，无隐式自动提交 |
| D2 | SQL 注入防护 | 使用参数化查询，无字符串拼接 SQL |
| D3 | N+1 查询 | 循环内无单条查询，批量操作使用 IN/Batch |
| D4 | 索引使用 | WHERE 条件字段有索引，无全表扫描风险 |
| D5 | 连接/事务资源释放 | defer rollback/commit，无连接泄漏 |

### E. 数据转换/序列化（类型断言/JSON/proto/映射）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| E1 | 类型断言安全 | 使用 comma-ok 模式，不直接 assert |
| E2 | 零值/nil 处理 | 转换前检查零值，nil 有显式分支 |
| E3 | 字段映射完整性 | 源和目标的字段一一对应，无遗漏/多余 |
| E4 | 序列化/反序列化一致性 | JSON tag 与结构体字段对应，无大小写陷阱 |

## 4. 单元测试反模式检测

**核心原则：测试审计不只是看"有没有测试"，而是看"测试是否真的在验证行为"。**

| # | 反模式 | 检测方法 | 严重性 |
|---|--------|---------|--------|
| T1 | **代入结果强行设计**：测试先写期望值，再让实现凑数 | 断言值是否有推导过程（注释/计算公式），纯魔法数字 → 可疑 | 高 |
| T2 | **Mock 返回硬编码真实业务值**：mock 的返回值恰好是生产数据的精确值 | mock 返回值是否来自常量/工厂函数，而非手写具体金额 | 高 |
| T3 | **断言恒真/恒假**：`assert(true)` 或 `assertNotNil(x)` 后 x 一定是非 nil | 检查断言条件是否可能为假，是否有负面测试 | 高 |
| T4 | **只覆盖 happy path**：所有测试用例都是正常输入 | 缺少：空输入、极大值、负数、nil、并发、超时 | 中 |
| T5 | **测试与实现耦合**：mock 验证了调用次数/参数顺序，而非行为结果 | 测试是否因为内部重构（不改行为）而失败 | 中 |
| T6 | **表格驱动但每行相同**：test cases 的输入差异不够 | 多个 case 的输入应覆盖不同等价类 | 低 |
| T7 | **setup 过重**：测试需要复杂的初始化链才能运行 | 测试应可独立运行，setup ≤ 10 行 | 低 |
| T8 | **跳过测试无理由**：`t.Skip()` 无注释说明原因 | 每个 skip 必须标注原因和恢复条件 | 中 |
| T9 | **测试修改全局状态**：测试间有隐式依赖顺序 | 每个测试应独立，不依赖执行顺序 | 高 |

## 5. 不确定项处理协议

**核心原则：不确定时必须问用户，禁止自行假设或跳过。**

### 收集规则

审计过程中遇到以下情况必须记录为不确定项：

| 情况 | 示例 |
|------|------|
| 业务规则不明确 | "这里应该是 VIP1 才能领取还是所有用户？" |
| 权衡无法判断 | "性能优化会影响可读性，哪个优先？" |
| 上下文不足 | "这个字段在其他地方是否也用了？" |
| 两种实现都合理 | "用 switch 还是 map 查找？" |

### 不确定项报告格式

```json
{
  "uncertainties": [
    {
      "id": "U1",
      "location": "file:line",
      "confidence": "high|medium|low",
      "description": "描述不确定的内容",
      "options": ["选项A", "选项B"],
      "recommendation": "推荐选项及理由",
      "impact": "选择不同选项的影响"
    }
  ]
}
```

### 主 agent 处理流程

1. 收到子 agent 报告后，逐个展示不确定项
2. 每项展示：位置 + 问题描述 + 可选项 + 推荐 + 影响
3. 等待用户选择：同意推荐 / 选择其他选项 / 跳过（标注"用户跳过"）
4. **禁止批量处理**：不可一次展示所有不确定项让用户"全部同意"

## 6. 注释完整性检查

**不要求每行都注释，但关键位置必须有注释。**

| 检查位置 | 要求 | 通过标准 |
|---------|------|---------|
| 公开函数/方法 | 有 godoc/docstring | 描述功能 + 参数含义 + 返回值 |
| 复杂算法 | 有行内注释 | 解释"为什么"而非"做什么" |
| 防御性代码 | 有原因注释 | `// 防止 XXX 情况` 而非裸 `if err != nil` |
| 魔法数字 | 有常量或注释 | `30 * time.Second` → `// HTTP 超时` 或常量 |
| 业务规则 | 有规则说明 | 金额计算/等级判定/权限判断处有注释说明规则来源 |
| TODO/FIXME | 有 issue 或截止日期 | 不能有无限期 TODO |

## 7. 审计报告格式

子 agent 必须返回以下 JSON 结构：

```json
{
  "audit_id": "唯一标识（时间戳+文件hash前8位）",
  "summary": {
    "total_findings": 0,
    "by_severity": {"critical": 0, "high": 0, "medium": 0, "low": 0},
    "by_dimension": {"bug": 0, "logic": 0, "comment": 0, "test": 0, "design": 0},
    "code_types_detected": ["金额", "并发", "DB"]
  },
  "findings": [
    {
      "id": "F1",
      "dimension": "bug",
      "severity": "high",
      "check_id": "A3",
      "location": "file:line",
      "code_snippet": "相关代码片段（≤5行）",
      "description": "问题描述",
      "suggestion": "修复建议",
      "confidence": "high|medium|low"
    }
  ],
  "uncertainties": [],
  "positive_notes": ["做得好的地方（可选，≤3条）"],
  "test_audit": {
    "test_files_found": true,
    "anti_patterns": [],
    "coverage_assessment": " adequate|weak|missing"
  },
  "comment_audit": {
    "missing_locations": [],
    "overall": " adequate|needs_improvement"
  }
}
```

### 严重性定义

| 级别 | 定义 | 行动 |
|------|------|------|
| critical | 必定导致错误/数据丢失/安全漏洞 | 阻塞交付，必须修复 |
| high | 高概率导致错误/不符合预期 | 强烈建议修复 |
| medium | 可能导致边界问题/可维护性差 | 建议修复 |
| low | 风格/最佳实践 | 可选修复 |

### 输出兜底（最低保障）

无论审计结果如何，报告必须包含：
- 至少 1 个 finding 或 positive_note（若代码质量优秀，total_findings 可为 0 并说明检查结果）
- 每个 finding 必须有 `check_id`（对应上述检查项编号）和 `location`（文件:行号）
- 不确定项必须标注 `confidence` 和 `impact`
- 测试审计和注释审计各至少 1 项评估

## 执行约束

- 子 agent **只读代码**，不修改任何文件
- 子 agent 不执行 bash 命令（纯静态分析）
- 审计范围仅限 git diff 涉及的文件和行（但检查上下文时允许读取调用方）
- 每个维度的检查项必须逐项执行，不可跳过
- 报告必须为完整 JSON，不可截断
- **不先入为主**：不可假设"这个项目质量很差"或"这个开发者不靠谱"，每项发现必须基于具体代码证据
- **不偷懒**：不可用"整体看起来不错"代替逐项检查，每个 check_id 必须有明确结论
