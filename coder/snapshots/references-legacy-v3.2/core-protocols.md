# Coder Core Protocols

SKILL.md 提供决策树骨架，本文件提供执行细节。

## 串联编排协议

```
for each S_k in chain [S1, S2, ..., Sn]:
  1. Pre:    收集语言上下文 + 前一阶段输出
  2. Inject: 注入语言 + 编辑规则 + 语义上下文 + 前置阶段结果
  3. Execute: 执行 S_k
  4. Verify: 语言验证链
  5. Gate:   验收（见下表）
  IF Gate FAIL → 最多重试 1 次
  IF 重试仍 FAIL → 停止后续阶段，向用户报告失败 + 诊断
```

每个阶段开始时输出 `[编排 {k}/{n}] {skill名} — {目标}`。

### 委托注入参数

| 字段 | 来源 |
|------|------|
| 语言 | 检测结果 |
| 编辑规则 | references/{prefix}-editing-rules.md |
| 语义上下文 | Pre 阶段收集的引用关系 |
| 测试模式 | references/{prefix}-conventions.md |
| 前置阶段结果 | 编排中上一阶段输出摘要 |

### 委托验收标准

| 检查项 | 合格标准 | 不合格处理 |
|--------|----------|-----------|
| 输出存在 | 产生了文件改动或明确结论 | [委托无输出]，回到本 skill |
| 验证链通过 | 语言验证链全部绿灯 | [验证失败]，修复后重试一次 |
| 目标达成 | 声明的目标已满足 | [目标未达成]，向用户报告差距 |
| 无副作用 | 未修改范围外文件（测试除外） | [副作用]，列出意外改动 |

### 路由失败恢复

| 场景 | 策略 |
|------|------|
| 目标 skill 未安装 | 回退通用编码 + 提示建议安装 |
| 多 skill 可能匹配 | 列出候选 + 适用场景，请用户选择 |
| 委托后执行失败 | 回到本 skill 继续语言专项 |

## 洞察发现（理解阶段主动扫描）

| 信号 | 检测方式 | 报告方式 |
|------|----------|----------|
| 相邻代码同类问题 | 语义工具发现同模式多处出现 | "发现 {N} 处同类问题，建议一并处理" |
| 死代码/未使用导出 | 引用数为零的公开符号 | "发现 {N} 个零引用符号，是否清理" |
| 风格不一致 | 与 conventions.md 对比 | "发现 {位置} 与惯例不符: {差异}" |
| 潜在风险 | 高耦合/深层嵌套/过长函数 | "注意: {函数} 复杂度较高，改动需谨慎" |
| 框架层异常 | 错误涉及类型解析/序列化/MCP | "框架差异: {正常} vs {报错}，根因: {分析}" |

**框架层错误专项**：涉及 MCP 工具/类型解析/序列化时：
1. `grep -r "同模式" 同项目` 找同参数模式
2. 对比正常 vs 报错的参数类型
3. 判定：框架自动解析 → 改注解；逻辑错误 → 修逻辑
4. 优先级：改注解/接口契约 > 加 isinstance 兼容层

## 经验持久化

### 加载历史经验（任务开始）

```
memory_search(query="{language} coding gotcha", limit=5)
```

找到经验后在步骤 1 末尾展示为 `⚠️ 历史经验提醒` 区块（≤1 行/条）。

### 标准化 tag 命名（v3.0 新增）

memory_store 时必须用以下 tag 命名（便于跨会话检索 + skill-evolver 跨进化追踪）：

| 经验类型 | tag 格式 | 例 |
|---------|---------|-----|
| 踩坑 | `coding-{lang}-gotcha` | `coding-go-gotcha`, `coding-python-gotcha` |
| 用户修正 | `coding-{lang}-gotcha` | （并入 gotcha，加 `correction` 子标记） |
| 编码惯例发现 | `coding-{lang}-convention` | `coding-go-convention` |
| 工具链命令 | `coding-{lang}-toolchain` | `coding-go-toolchain` |
| 审计模式 | `coding-{lang}-audit` | `coding-python-audit` |

`{lang}` 必须小写：`go` / `python` / `typescript` / `rust`。

**反例**（PP-20 重现风险）：禁止用 `coder-Python`、`feedback-Python` 等不一致 tag——会导致 `memory_search(query="python coding gotcha")` 返回空。

### 保存新经验（任务结束，SKILL.md 经验总结阶段触发）

| 触发条件 | 存储内容 | tag（v3.0 标准） |
|----------|----------|------------------|
| 踩坑（工具报错/降级/重试 2+ 次） | "踩坑: {描述}。正确做法: {做法}" | `coding-{lang}-gotcha` |
| 用户修正（用户纠正过做法 ≥1 次） | "修正: {原始做法} → {正确做法}" | `coding-{lang}-gotcha` |
| 发现模式（重复代码结构） | "发现: {语言/框架} 项目中 {模式} 的规律" | `coding-{lang}-convention` |
| 工具链配置（版本/环境问题） | "环境: {工具} {版本} 在 {条件} 下的行为" | `coding-{lang}-toolchain` |
| 工具链发现（新探测到的构建命令） | "环境: {工具} 关键命令: {命令列表}" | `coding-{lang}-toolchain` |

**去重**：存储前 `memory_search(query="{关键词}", limit=3)`，核心描述重合 >50% 则更新而非新建。

## 保护测试（仅完整路径）

```
2.5a. 识别关键行为锚点
2.5b. 编写参数化测试 — Go: table-driven, Python: pytest+parametrize
2.5c. 确认全部 PASS
```

Fallback：

| 场景 | 处理 |
|------|------|
| 私有符号无测试入口 | 标注 [未保护] |
| 依赖外部服务 | 标注 [未保护-外部依赖] |
| 改动 >10 个函数 | 按调用链分组测试 |
| 全部 Fallback 触发 | **必须用户确认** |

## 汇报格式

```markdown
## 改动摘要
- 语言：{语言} | 文件数：{N} | 类型：{类型} | 风险：{等级}

## 编排过程（如有）
- [阶段1] {skill}: {结果摘要}

## 洞察发现
- {发现}

## 保护测试
- 文件：{path}（或"跳过"） | 用例：{N} | 回归：PASS / {M}P {K}F

## 审查结果
- {审查要点摘要}

## 文件列表
- path/to/file: {改动描述}

## 验证结果
- lint: PASS/FAIL | type: PASS/FAIL | test: {M}P/{K}F
```

汇报完成后，检查是否产生可持久化经验，自动保存。
