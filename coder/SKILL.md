# Coder — 多语言编码编排层

## DO NOT trigger

| 条件 | 行为 |
|------|------|
| 无匹配的项目语言文件且目标文件无法确定语言 | 退出，使用通用编码能力 |
| 纯文档任务（写规范文档、README、变更日志） | 退出，使用文档 skill 或通用文本能力 |
| 非 coding 任务（分析数据、翻译、做计划） | 退出，使用通用能力 |
| 用户明确说"不用语言相关工具" | 尊重意图，降级到 grep/Read 模式 |

## 语言检测

激活后，按以下优先级检测项目语言：

**检查 1：目标文件扩展名**（最高优先级）
- 用户指定了要修改的文件 → 从扩展名直接确定语言

**检查 2：项目文件探测**

| 项目文件 | 语言 | references 前缀 |
|----------|------|-----------------|
| `go.mod` | Go | `go-` |
| `pyproject.toml` / `setup.py` / `setup.cfg` / `requirements.txt` | Python | `python-` |
| 未匹配 | 退出，提示"不支持的语言" | - |

**检测到多种语言时**：优先匹配目标文件扩展名。仍无法确定 → 列出候选语言让用户选择。

## 语言特定铁律加载

检测到语言后，用 Read 加载 `references/` 中对应的文件（**强制，非可选**）：

| 加载时机 | 文件模式 | 说明 |
|----------|----------|------|
| **步骤 1 前** | `references/{prefix}-tooling.md` | 工具选择策略、语义导航优先级 |
| **步骤 3 前** | `references/{prefix}-editing-rules.md` | 编辑陷阱、文件归属、格式化规则 |
| **步骤 4 前** | `references/{prefix}-verification-loop.md` | 验证链、预存失败识别 |
| 新增/重组代码时 | `references/{prefix}-conventions.md` | 编码惯例、架构模式、代码风格 |

当前支持的语言 references：

| 语言 | 工具策略 | 编辑规则 | 验证循环 | 编码惯例 |
|------|----------|----------|----------|----------|
| Go | `go-gopls-strategy` | `go-editing-traps` | `go-verification-loop` | `go-conventions` |
| Python | `python-tooling` | `python-editing-rules` | `python-verification-loop` | `python-conventions` |

## 前置条件检查（按语言）

检测到语言后，执行对应的预检：

**Go**:
1. `ls go.mod` — 不存在则确认是否 Go 项目
2. 尝试 `go_workspace` — 不可用则降级到 grep/Read

**Python**:
1. `ls pyproject.toml setup.py requirements.txt` — 确认 Python 项目
2. 检测虚拟环境：`.venv/` 或 `venv/`
3. 确认 `uv` 可用：`uv --version`

## 任务复杂度分级

| 等级 | 判定条件 | 流程 |
|------|----------|------|
| **简单** | 纯文件删除、注释修改、常量删除、配置格式调整 | **快速路径**：1→3→4→5，跳过步骤 2 和 2.5 |
| **标准** | 新增函数/字段、修改业务逻辑、同包重构 | **标准流程**：完整 6 步 |
| **复杂** | 金额计算、并发安全、公开 API 签名变更、跨包移动 | **完整流程**：含步骤 2.5 保护测试 |

### 快速路径（简单任务专用）

```
步骤 1: 语义工具定位所有引用 → 生成改动清单
步骤 3: 按清单逐文件改动
步骤 4: 语言验证链（见 references/{prefix}-verification-loop.md）
步骤 5: 改动摘要 + 验证结果
```

## 定位

本 skill 是多语言项目的编码入口路由器：

1. **语言检测** → 确定项目语言
2. **上下文加载** → 读取对应的 references 文件
3. **意图识别** → 路由到正确执行路径
4. **验证门控** → 确保每次改动通过语言特定验证链

| 本 skill 提供 | 委托给其他 skill |
|--------------|-----------------|
| 语言检测 + 上下文加载 | tdd — 测试驱动开发流程 |
| 语言特定工具选择策略 | simplify — 行为不变简化 |
| 语言特定编辑规则 | code-review — diff 审查 |
| 语言特定验证链 | unit-testing-best-practices — 测试设计 |
| 语义感知的重构方法论 | diagnose — bug 诊断循环 |

## 任务路由表

| 用户意图 | 执行路径 | 加载 references |
|----------|----------|-----------------|
| 重构/重组/重命名 | **本 skill 主导** → 阶段式重构流程 | tooling, editing-rules, verification-loop |
| 优化/简化 | **simplify** + 本 skill 验证门控 | verification-loop |
| 新功能 | **tdd** + 本 skill 语义导航 | tooling |
| 写测试/设计测试 | **unit-testing-best-practices** | tooling |
| 修 bug | **diagnose** + 本 skill 语义导航 | tooling, verification-loop |
| 审查代码 | **code-review** | verification-loop |
| 混合意图 | 先完成语言专项部分，再委托对应 skill | 按需加载 |
| 意图不明确 | 列出最多 3 条可能路径让用户选择 | - |

### 路由失败恢复

| 失败场景 | 恢复策略 |
|----------|----------|
| 目标 skill 未安装 | 回退通用编码模式 + 提示建议安装 |
| 多个 skill 可能匹配 | 列出候选 + 适用场景，请用户选择 |
| 委托后 skill 执行失败 | 回到本 skill 继续语言专项部分，不中止整体任务 |

## 委托协议

委托给其他 skill 前，执行三步注入：

```
1. Pre:    用语义工具收集语言特定上下文
2. Inject: 在调用 prompt 中注入收集到的上下文参数 + 编辑规则
3. Post:   运行语言验证链确认改动未破坏构建
```

### Go 注入参数表

| 委托目标 | Pre | Inject | Post |
|----------|-----|--------|------|
| `/simplify` | `go_symbol_references` 获取引用数 | "引用数 N，Go 文件用 Write 不用 Edit" | `go_diagnostics` → `go build` → `go test` |
| `/tdd` | `go_package_api` 提取签名；`go_search` 检查已有实现 | "API 签名列表，Go 测试用 table-driven + testify" | `go_diagnostics` → `go build` → `go test` |
| `/unit-testing-best-practices` | `go_package_api` 提取被测签名 | "被测签名列表，Go mock 用 mockey，测试文件 {name}_test.go" | `go test -v` |
| `/diagnose` | `go_search` + `go_symbol_references` 追踪调用链 | "错误 msg，调用链列表，最小复现 go test -run" | `go test -run` |
| `/code-review` | `git diff --stat` + `go_diagnostics` | "改动文件列表，Go 检查：gerror+decimal+ctx+Transaction" | `go_diagnostics` → `go vet` |

### Python 注入参数表

| 委托目标 | Pre | Inject | Post |
|----------|-----|--------|------|
| `/simplify` | grep + pyright 定位引用数 | "引用数 N，编辑后必须 ruff format + ruff check" | `ruff check` → `pyright` → `pytest` |
| `/tdd` | `pyright` 分析模块 API | "目标 API 签名，Python 测试用 pytest + parametrize" | `ruff check` → `pyright` → `pytest` |
| `/unit-testing-best-practices` | Read 被测模块理解接口 | "被测接口列表，Python mock 用 unittest.mock，测试文件 test_{name}.py" | `pytest -v` |
| `/diagnose` | grep 追踪调用链 + 错误位置 | "错误 msg，调用链列表，最小复现 pytest {file}::{test}" | `pytest -k` |
| `/code-review` | `git diff --stat` + `ruff check` | "改动文件列表，Python 检查：类型注解+异常处理+ruff 合规" | `ruff check` → `ruff format --check` → `pyright` |

## 通用执行骨架（标准/复杂任务）

标准&复杂任务共用 6 步流程。**简单任务走快速路径。**

**步骤 2 必须等用户确认后才可执行。复杂任务必须经过步骤 2.5 保护测试。**

| 步骤 | 做什么 | 输出物 | 验证门 |
|------|--------|--------|--------|
| **1. 理解** | 语义工具导航：搞清楚改什么、影响谁 | 改动清单 + 影响范围 + 风险评级 | 清单条目 > 0 |
| **2. 方案** | 告诉用户要改什么、为什么、风险等级 | diff 预览 + 改动说明 + 风险评级 + 保护测试计划 | **用户确认** |
| **2.5 保护测试** | 复杂任务先写测试锁定当前行为 | 测试文件 + 测试全部 PASS | 测试全绿 |
| **3. 执行** | 分步改动（遵循语言编辑规则），批量验证 | 逐步改动结果 | 全部改动完成后一次性验证 |
| **4. 验证** | 语言验证链（含保护测试回归） | 通过/失败 + 失败归因 | 构建通过 + 保护测试全绿 |
| **5. 汇报** | 改了什么、为什么、验证结果 | 文件列表 + 每文件改动摘要 + 验证结果 | 包含文件列表 |

### 步骤 1：理解（只读，语义导航）

**打开每个源文件前，先用语义工具回答 3 个问题：**
1. 这个符号被谁引用？
2. 这个模块有什么公开 API？
3. 有没有同名/相似符号？

然后才用 `ctx_batch_execute` 批量 grep 做补充（不重复语义工具已覆盖的内容）。

| 任务类型 | 语义操作 | 必须回答的问题 |
|----------|----------|---------------|
| 重构 | API 提取 + 引用查找 + Read | 改什么？引用数多少？风险等级？ |
| 新需求 | API 提取 + 搜索已有实现 + 依赖分析 | 目标 API？已有相关实现？依赖关系？ |
| Bug 修复 | 错误搜索 + 引用追踪 + Read | 错误位置？调用链？根因假设？ |
| 简化/删除 | 引用查找 + 搜索 | 引用数？归属文件？可安全删除？ |
| 审查 | `git diff --stat` + 诊断 | 改了哪些文件？构建状态？ |

### 步骤 2：方案（**必须用户确认**，快速路径跳过）

展示方案，等用户确认后才执行：

```
1. 改动范围：涉及哪些模块/文件/函数
2. 改动内容：每个文件要做什么（新增/修改/删除/重命名）
3. 风险评级：简单/标准/复杂 + 理由
4. 验证计划：用哪些命令验证
5. 保护测试计划（复杂任务时）：锁定哪些行为、测试文件路径、用例数量
```

**风险评级**（满足任一即升级）：

| 风险等级 | 触发条件 |
|----------|----------|
| **复杂** | 金额/精度计算、并发安全改动、公开 API 签名变更 |
| **标准** | 流程重组、跨模块移动、数据库/事务相关改动 |
| **简单** | 纯删除/注释、同模块内重命名、新增纯内部函数、修改文本 |

### 步骤 2.5：保护测试（仅复杂任务）

跳过条件：简单/标准任务明确标注跳过。

```
2.5a. 识别关键行为锚点 — 提取将被影响的所有公开函数/方法
2.5b. 编写参数化保护测试 — 锁定改动前行为
2.5c. 确认测试通过 — 全部 PASS 后继续
```

**Go 保护测试**：`{name}_guard_test.go`，table-driven 结构
**Python 保护测试**：`tests/test_{name}_guard.py`，pytest + parametrize 结构

**无法写测试的 Fallback**：

| 场景 | 处理方式 |
|------|----------|
| 私有符号且无测试入口 | 标注 [未保护]，建议用户审查 |
| 依赖外部服务无法 mock | 标注 [未保护-外部依赖] |
| 改动超过 10 个函数 | 按调用链分组，每组独立保护测试 |
| **所有 Fallback 都触发** | 标注 [未保护-全部]，**必须用户确认后才继续** |

### 步骤 3：执行（分步，遵循语言编辑规则，批量验证）

- **先完成所有文件改动**，再一次性运行验证
- **不要每改动一个文件就验证一次**——批量验证，只对错误做增量修复

根据任务类型走差异化子流程：

#### 重构子流程

加载 `references/{prefix}-editing-rules.md`，按风险从低到高排序执行：

| 风险 | 操作 | 验证 |
|------|------|------|
| 零 | 删除零引用标识符 | lint 检查 |
| 低 | 同模块文件拆分/合并/重命名 | 构建 |
| 中 | 跨模块移动 | 构建 + 改导入 |
| 高 | 公开 API 签名变更 | 构建 + 测试（含保护测试） |

#### 新需求子流程

委托 `/tdd`，注入语言上下文后执行 TDD 循环。

#### Bug 修复子流程

委托 `/diagnose`，注入语言上下文后执行。

#### 简化/删除子流程

委托 `/simplify`，注入引用安全上下文后执行。

#### 审查子流程

委托 `/code-review`，注入语言检查清单后执行。

### 步骤 4：验证

加载 `references/{prefix}-verification-loop.md`，执行完整验证链。

**保护测试回归验证**（步骤 2.5 产出的测试）：
- 运行保护测试确认全部 PASS
- 保护测试失败 → 标记为行为变更，需向用户确认是否为预期变更

测试失败时：
1. `git log --oneline -5 -- path/to/test` 检查是否预存问题
2. 预存问题 → 标注并继续
3. 新引入失败 → 回退到步骤 3，修复后重新验证

### 步骤 5：汇报

```
## 改动摘要
- 语言：{检测到的语言}
- 改动文件：{N} 个
- 改动类型：{重构/新增/修复/简化/删除}
- 风险等级：{简单/标准/复杂}

## 保护测试结果（步骤 2.5）
- 保护测试文件：{path}（或"跳过"）
- 用例总数：{N}
- 回归结果：全部 PASS / {M} PASS {K} FAIL（需确认）

## 文件列表
- path/to/file: {改动描述}

## 验证结果
- lint:     ✅ / ❌
- type:     ✅ / ❌
- test:     ✅ {M passed} / ❌ {N failed}（预存 {X}）
```

## 输出保障

执行过程中每次输出包含：

1. **当前步骤**：理解 / 方案 / 保护测试 / 执行 / 验证 / 汇报
2. **语言**：Go / Python / ...
3. **已委托的 skill**：无 / {skill 名}
4. **工具模式**：语义工具正常 / 降级
5. **风险等级**：简单 / 标准 / 复杂
