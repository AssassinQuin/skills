# Coder — 多语言编码元技能

## DO NOT trigger

| 条件 | 行为 |
|------|------|
| 无匹配语言且目标文件无法确定语言 | 退出，使用通用编码能力 |
| 纯文档任务（README、规范文档、变更日志） | 退出，使用文档 skill |
| 非 coding 任务（分析数据、翻译、做计划） | 退出，使用通用能力 |
| 用户明确说"不用语言相关工具" | 尊重意图，降级到 grep/Read |

## 语言检测

**检查 1：目标文件扩展名**（最高优先级）→ 直接确定语言

**检查 2：项目文件探测**

| 项目文件 | 语言 | references 前缀 | 验证链 |
|----------|------|-----------------|--------|
| `go.mod` | Go | `go-` | `go_diagnostics` → `go build` → `go test` → `go vet` |
| `pyproject.toml` / `setup.py` / `requirements.txt` | Python | `python-` | `ruff check` → `pyright` → `pytest` |
| 未匹配 | 退出 | - | - |

多语言歧义 → 优先匹配目标文件扩展名 → 仍不确定 → 列出候选让用户选择。

## 上下文加载（强制）

检测到语言后，用 Read 加载 references（**不可跳过**）：

| 时机 | 文件 | 内容 |
|------|------|------|
| 步骤 1 前 | `references/{prefix}-tooling.md` | 语义工具选择、导航策略 |
| 步骤 3 前 | `references/{prefix}-editing-rules.md` | 编辑规则、文件归属、格式化 |
| 步骤 4 前 | `references/{prefix}-verification-loop.md` | 验证链、预存失败识别 |
| 新增/重组代码时 | `references/{prefix}-conventions.md` | 编码惯例、架构模式 |

## 前置条件检查（按语言）

**Go**: `ls go.mod` → 尝试 `go_workspace`（不可用则降级）
**Python**: `ls pyproject.toml` → 检测 `.venv/` → `uv --version`

## 经验持久化（memory）

### 任务开始时：加载历史经验

```
memory_search(query="{language} coding gotcha {语言} 踩坑", tags=["coder-{语言}"], limit=5)
```

将搜索到的经验作为**提醒清单**注入到步骤 1 的理解阶段。如果发现相关经验：
- 在步骤 1 输出中追加 "⚠️ 历史经验提醒" 区块
- 列出每条经验的要点（≤1 行/条）

### 任务结束时：保存新经验

每次任务完成（步骤 5 之后），检查是否产生了值得保存的经验：

| 触发条件 | 存储内容 | tags |
|----------|----------|------|
| 踩坑（工具返回错误、降级到 grep/sed、重试 2+ 次才成功、格式化命令失败） | "踩坑: {描述}。正确做法: {做法}" | `coder-{语言},gotcha` |
| 发现模式（重复出现的代码结构/问题） | "发现: {语言} 项目中 {模式} 的规律" | `coder-{语言},pattern` |
| 工具链配置（特定版本/环境问题） | "环境: {工具} {版本} 在 {条件} 下的行为" | `coder-{语言},tooling` |

**去重**：存储前 `memory_search(query="{要点关键词}", limit=3)`，若任一条结果的核心描述与待存储内容要点重合 >50% 则更新而非新建。

**目的**：防止上下文污染——历史经验不自动注入每次对话，只在 coder 激活时按语言精确加载。

## 任务复杂度分级

| 等级 | 判定条件 | 流程 |
|------|----------|------|
| **简单** | 纯删除/注释/常量/配置格式调整 | **快速路径**：1→3→4→5 |
| **标准** | 新增函数/修改逻辑/同包重构 | **标准流程**：完整 6 步 |
| **复杂** | 金额计算/并发/API 签名变更/跨模块移动/**框架交互**（MCP/ORM/序列化框架行为差异） | **完整流程**：含步骤 2.5 |

## 编排模式

用户意图可能需要单个或多个 skill 协作。根据意图识别编排模式：

### 模式判定表

| 用户意图 | 编排模式 | skill 链 | 加载 references |
|----------|----------|----------|-----------------|
| 重构/重组/重命名 | **自主导导** | 本 skill → (可选 simplify) | tooling, editing-rules, verification-loop |
| 优化/简化 | **单 skill** | simplify | verification-loop |
| 新功能 | **单 skill** | tdd | tooling |
| 写测试 | **单 skill** | unit-testing-best-practices | tooling |
| 修 bug | **单 skill** | diagnose | tooling, verification-loop |
| 审查代码 | **单 skill** | code-review | verification-loop |
| 诊断+修复+补测试 | **串联** | diagnose → 执行修复 → tdd | tooling, verification-loop |
| 审查+修复 | **串联** | code-review → diagnose → 执行修复 | verification-loop, tooling |
| 重构+简化+补测试 | **串联** | 执行重构 → simplify → tdd | tooling, editing-rules, verification-loop |
| 混合/不明确 | **交互确认** | 列出 ≤3 条候选路径让用户选择 | - |

### 串联编排协议

当需要多个 skill 串联时，每个 skill 作为编排中的一个阶段：

```
对于 skill 链 [S1, S2, ..., Sn]:
  for each S_k in chain:
    1. Pre:    收集语言上下文 + 前一阶段的输出（如有）
    2. Inject: 注入上下文 + 前置阶段结果摘要
    3. Execute: 执行 S_k
    4. Verify: 语言验证链
    5. Gate:   验收（见下方验收标准）
    IF Gate FAIL → 最多重试 1 次（修复后重新 Execute→Verify→Gate）
    IF 重试后仍 FAIL → 停止后续阶段，向用户报告失败 + 诊断
```

**编排状态追踪**：每个阶段开始时输出 `[编排 {k}/{n}] {skill名} — {目标}`

### 委托注入参数（通用模板）

每个委托阶段按以下模板注入（具体命令来自 references/{prefix}-tooling.md）：

| 字段 | 来源 | 说明 |
|------|------|------|
| 语言 | 检测结果 | "当前项目语言: {Go/Python}" |
| 编辑规则 | references/{prefix}-editing-rules.md | "编辑后必须 {格式化命令}" |
| 语义上下文 | Pre 阶段收集 | "引用数 N，调用链: {列表}" |
| 测试模式 | references/{prefix}-conventions.md | "测试框架: {pytest/table-driven}" |
| 前置阶段结果 | 编排中上一阶段输出 | "上一阶段 {skill} 发现: {摘要}" |

### 委托验收标准

每个委托 skill 执行后，检查以下条件：

| 检查项 | 合格标准 | 不合格处理 |
|--------|----------|-----------|
| **输出存在** | skill 产生了文件改动或明确结论 | 标注 [委托无输出]，回到本 skill 继续 |
| **验证链通过** | 语言验证链全部绿灯 | 标注 [验证失败]，修复后重试一次 |
| **目标达成** | 委托前声明的目标已满足 | 标注 [目标未达成]，向用户报告差距 |
| **无副作用** | 未修改目标范围外的文件（除测试文件） | 标注 [副作用]，列出意外改动 |

验收不合格 → 停止后续阶段，汇报当前状态让用户决策。

## 路由失败恢复

| 失败场景 | 恢复策略 |
|----------|----------|
| 目标 skill 未安装 | 回退通用编码模式 + 提示建议安装 |
| 多个 skill 可能匹配 | 列出候选 + 适用场景，请用户选择 |
| 委托后 skill 执行失败 | 回到本 skill 继续语言专项部分 |

## 通用执行骨架（标准/复杂任务）

**简单任务走快速路径**（1→3→4→5）。步骤 2 必须等用户确认。复杂任务必须经过步骤 2.5。

| 步骤 | 做什么 | 输出物 | 验证门 |
|------|--------|--------|--------|
| **1. 理解** | 语义导航 + 加载历史经验 | 改动清单 + 影响范围 + 风险评级 + 洞察发现 | 清单 > 0 |
| **2. 方案** | 展示改动方案 + 编排计划 | diff 预览 + 编排阶段 + 风险评级 | **用户确认** |
| **2.5 保护测试** | 复杂任务锁定当前行为 | 测试文件 + 全部 PASS | 测试全绿 |
| **3. 执行** | 按编排模式分步执行 | 逐步改动 + 编排阶段状态 | 全部完成后一次性验证 |
| **4. 验证** | 语言验证链 + 保护测试回归 | 通过/失败 + 失败归因 | 构建通过 + 保护测试全绿 |
| **5. 汇报** | 改动摘要 + 洞察发现 + 经验持久化 | 结构化汇报 | 包含文件列表 |

### 步骤 1：理解（只读 + 洞察发现）

**语义导航**：打开源文件前，先用语义工具回答：
1. 这个符号被谁引用？
2. 这个模块有什么公开 API？
3. 有没有同名/相似符号？

然后才用 `ctx_batch_execute` 补充（不重复语义工具已覆盖的内容）。

**洞察发现**（主动扫描，不等用户要求）：

理解阶段除了回答用户直接问题，还必须扫描以下信号：

| 信号 | 检测方式 | 报告方式 |
|------|----------|----------|
| 相邻代码同类问题 | 语义工具发现同模式的多处出现 | "发现 {N} 处同类问题，建议一并处理" |
| 死代码/未使用导出 | 引用数为零的公开符号 | "发现 {N} 个零引用符号，是否清理" |
| 风格不一致 | 与 references/{prefix}-conventions.md 对比 | "发现 {位置} 与惯例不符: {差异}" |
| 潜在风险 | 高耦合/深层嵌套/过长函数 | "注意: {函数} 复杂度较高，改动需谨慎" |
| 框架层异常 | 错误涉及类型解析/序列化/MCP 工具 → 对比同项目正常工作的代码 | "⚠️ 框架差异: {正常代码} vs {报错代码}，根因: {分析}" |

**框架层错误专项**：当错误涉及 MCP 工具、类型自动解析、序列化时，必须：(1) 检查同项目其他工具是否有相同模式；(2) 对比正常代码与报错代码的差异；(3) 定位是框架行为差异还是代码逻辑差异；(4) 优先修改接口契约（类型注解/参数传递），而非添加兼容层。

洞察在步骤 1 输出的 **"洞察发现"** 区块中呈现，由用户决定是否纳入步骤 2 方案。

**历史经验提醒**：从 memory 加载的踩坑记录在此步骤末尾展示为 `⚠️ 提醒` 区块。

### 步骤 2：方案（用户确认，快速路径跳过）

展示方案：

```
1. 改动范围 + 涉及模块/文件/函数
2. 改动内容（新增/修改/删除/重命名）
3. 编排计划（如多 skill 串联）：[阶段1] → [阶段2] → ...
4. 风险评级：简单/标准/复杂 + 理由
5. 验证计划 + 保护测试计划（复杂任务时）
```

风险评级（满足任一即升级）：

| 等级 | 触发条件 |
|------|----------|
| **复杂** | 金额/精度计算、并发安全、公开 API 签名变更 |
| **标准** | 流程重组、跨模块移动、数据库/事务相关 |
| **简单** | 纯删除/注释、同模块重命名、新增纯内部函数、文本修改 |

### 步骤 2.5：保护测试（仅复杂任务）

```
2.5a. 识别关键行为锚点
2.5b. 编写参数化保护测试 — Go: table-driven, Python: pytest+parametrize
2.5c. 确认全部 PASS
```

无法写测试的 Fallback：

| 场景 | 处理 |
|------|------|
| 私有符号无测试入口 | 标注 [未保护]，建议用户审查 |
| 依赖外部服务 | 标注 [未保护-外部依赖] |
| 改动 >10 个函数 | 按调用链分组，每组独立测试 |
| **全部 Fallback 触发** | 标注 [未保护-全部]，**必须用户确认** |

### 步骤 3：执行（按编排模式，批量验证）

- **先完成所有文件改动**，再一次性运行验证
- 不每改动一个文件就验证——批量验证，只对错误做增量修复
- 串联编排中，每个阶段结束后运行语言验证链作为门控

子流程（按任务类型）：

| 任务类型 | 流程 | 附加 |
|----------|------|------|
| 重构 | 本 skill 主导，按风险从低到高 | 加载 editing-rules |
| 新需求 | 委托 `/tdd` + 注入语言上下文 | TDD 循环 |
| Bug 修复 | 委托 `/diagnose` + 注入调用链 | 定位→修复→回归测试 |
| 简化/删除 | 委托 `/simplify` + 注入引用安全 | 逐点简化 |
| 审查 | 委托 `/code-review` + 注入检查清单 | 逐文件审查 |

### 步骤 4：验证

加载 `references/{prefix}-verification-loop.md`，执行完整验证链。

**MCP/框架工具专项**：修复涉及 MCP 工具时，必须直接调用工具验证（非仅导入测试），并检查同项目其他工具是否受影响。

保护测试回归：运行保护测试 → 失败则标记为行为变更，向用户确认。

测试失败处理：
1. `git log --oneline -5 -- path/to/test` → 预存问题标注并继续
2. 新引入失败 → 回退步骤 3 修复（**最多循环 2 次**，第 3 次仍失败则终止并汇报）

### 步骤 5：汇报 + 经验持久化

```
## 改动摘要
- 语言：{检测到的语言}
- 改动文件：{N} 个
- 改动类型：{重构/新增/修复/简化/删除}
- 风险等级：{简单/标准/复杂}

## 编排过程（如有）
- [阶段1] {skill}: {结果摘要}
- [阶段2] {skill}: {结果摘要}

## 洞察发现
- {发现1}
- {发现2}

## 保护测试结果
- 保护测试文件：{path}（或"跳过"）
- 用例总数：{N} | 回归结果：全部 PASS / {M}P {K}F

## 文件列表
- path/to/file: {改动描述}

## 验证结果
- lint: ✅/❌ | type: ✅/❌ | test: ✅ {M}P / ❌ {N}F（预存 {X}）
```

汇报完成后，检查是否产生可持久化的经验（见"经验持久化"一节），自动保存到 memory。

## 输出保障

每次输出包含：

1. **当前步骤**：理解 / 方案 / 保护测试 / 执行 / 验证 / 汇报
2. **语言**：Go / Python / ...
3. **编排阶段**：无 / [{k}/{n}] {skill名}
4. **工具模式**：语义工具正常 / 降级
5. **风险等级**：简单 / 标准 / 复杂
