# Coder v7.0 — Skill 模块化设计

**日期**：2026-06-25
**前置**：v6.3（13 Phase + 14 hook + 22 references + token 经济）
**来源**：Anthropic Cybersecurity Skills 的 skill anatomy（独立目录 + scripts + assets）
**状态**：设计完成，待实施

---

## 0. v7.0 核心目标

**从"平铺 references"升级为"模块化 skills"**，每个 Phase / 协议模块成为可独立发现 + 配套脚本 + 配套资源的子 skill。

### 与 v6.3 的对比

| 维度 | v6.3 | v7.0 |
|---|---|---|
| references 组织 | `coder/references/*.md` 平铺 | `coder/modules/{name}/` 每个独立目录 |
| 脚本归属 | `coder/scripts/` 集中 | 高频脚本配套到对应 module + 全局共享入口 |
| 资源归属 | `coder/templates/` 集中 | 模板配套到对应 module + 全局共享 |
| 可发现性 | orchestrator 按文件名猜 | 每个 module 是独立 skill（SKILL.md frontmatter） |
| 分发 | 整个 coder 包 | 单个 module 可独立分发（未来） |
| Skill anatomy 兼容 | ❌ | ✅ 符合 cybersec 标准 |

## 2. Skill Anatomy 标准（每个 module）

每个 module 是独立 skill，遵循 cybersec anatomy：

```yaml
---
# modules/{name}/SKILL.md frontmatter
name: {module-name}                   # kebab-case, 1-64 chars
description: >-                       # keyword-rich for discovery
  {keyword-heavy description}
domain: coding                        # 全部属 coding domain
subdomain: {phase|protocol|mcp|strategy}  # 子域分类
phase: {0|0.5|0.6|1|2.5|3|4|4.5|5|6|7|-1}  # 哪个 Phase（如有）
tags: [...]                           # discoverable tags
tokens_estimate: {N}                  # 完整加载成本
load_priority: {always|high|on-demand}
load_when: "{何时加载}"
keywords: {keyword string}
version: "1.0"
parent_skill: coder                   # 父 skill
license: Apache-2.0
---

# {Module Title}

## Workflow / 协议
{主要内容}

## Verification
{如何确认本 module 成功应用}

## Anti-pattern
{反模式}

## 引用
{上下游 module}
```

## 4. 兼容性策略

### 4.1 references/ 软链接（v7.0 → v7.2 渐进废弃）

迁移期间保留 `coder/references/` 目录，但内容改为指向 `modules/` 的软链接：

```bash
# 例如
references/phase-0-intent-capture.md → ../modules/phase-0-intent-capture/SKILL.md
references/hard-constraints.md → ../modules/hard-constraints/SKILL.md
```

**目的**：现有外部引用（其他 skill / 文档）不断。

### 4.2 v7.2 完全移除 references/

迁移完成 + 所有引用更新后，删除 `references/` 目录。

### 4.3 scripts/ 保留全局入口

`scripts/coder-state.{sh,py}` 和 `scripts/init-project.{sh,py}` 是跨多 module 共享，**保留在 `coder/scripts/`**。

`scripts/validate-delivery.py` 移到 `modules/phase-4.5-delivery-validation/scripts/`，但 `scripts/` 保留软链接兼容现有 init-project.py 引用。

## 8. Tradeoffs

### 8.1 模块化 vs 平铺

**模块化**（v7.0）→ 可独立发现 + 配套脚本绑定 + 符合 cybersec 标准
**平铺**（v6.3）→ 引用路径短 + 文件少 + 简单

**当前选择**：模块化（v7.0）—— 长期收益 > 短期复杂度。

### 8.2 兼容期长度

**短**（v7.0 → v7.1 移除 references/）→ 干净但可能破坏外部引用
**长**（v7.0 → v7.2/v7.3）→ 安全但代码库臃肿

**当前选择**：v7.0 → v7.2（保留 references/ 软链接 2 个版本）。

### 8.3 scripts/ 归属

**全部移到 modules/**（极端模块化）→ 强绑定但跨 module 调用难
**全部留在 scripts/**（v6.3 现状）→ 共享方便但脚本与文档脱节
**混合**（v7.0 选择）→ module 专属脚本进 module + 跨 module 共享留 scripts/

**当前选择**：混合。

## 9. 风险

| 风险 | 缓解 |
|---|---|
| 引用路径大量断 | references/ 软链接兼容期 |
| orchestrator 找不到 module | SKILL.md frontmatter `modules:` 列表 + `subskill:` 字段 |
| module 间循环引用 | modules 不引用其他 module（只在 SKILL.md 主 router 引用） |
| 脚本路径变化破坏 init-project.py | scripts/ 保留软链接 + 全局入口 |
| 升级成本（用户重新跑 init） | init 兼容 v6.3 项目（无强制升级） |

## 11. 总结

**v7.0 = v6.3 + skill 模块化**：
- 22 个 references → 22 个 modules（每个独立目录 + SKILL.md + scripts/ + assets/）
- 主 SKILL.md 仍是 router，但引用 `modules/`
- references/ 保留软链接（兼容期）
- scripts/ 保留全局共享入口

**收益**：
- 每个 Phase module 可独立发现 / 独立分发 / 独立升级
- 脚本与文档强绑定（validate-delivery.py 在 phase-4.5 module 内）
- 100% 符合 cybersec skill anatomy
- 为未来"独立打包分发单 module"打基础

**代价**：
- 目录结构嵌套深一层
- 引用路径变长（但 SKILL.md frontmatter `modules:` 列表可发现）

**实施分两阶段**：
- v7.0 P0：5 核心 module（高频 + 需配套脚本）
- v7.1 P1：剩余 17 module（纯文档为主）
