# Cognitive Apprenticeship（认知学徒制四要素）

来源：Collins/Brown/Newman 1989 → Koblic 2024 应用于 AI prompt。

让 AI 从"答案机"升级为"导师"。四要素写进 SKILL.md。

---

## 四要素定义

### 1. Modeling（专家出声思考）

**核心**：把专家的隐性判断显式化。

**问题模板**：
- "专家这一步会想什么？"
- "专家为什么选 A 不选 B？"
- "专家在这一步会想到哪些边角案例？"
- "如果错了，专家会怎么回退？"

**SKILL.md 写法**：

```
3. 加载历史经验：memory_search(query="{language} coding gotcha", limit=5)

> 💡 专家视角：limit=5 不是随便选的——超过 5 条 agent 注意力会分散，
> 反而漏掉最相关的那条。如果 memory_search 返回 8 条，先看创建时间
> 最近 3 条 + 相关度最高 2 条。如果返回 0 条，标注 "⚠️ 无历史经验"，
> 提醒用户首次踩坑风险。
```

### 2. Coaching（练习时反馈）

**核心**：在 agent 执行过程中，模拟专家的反馈。

**问题模板**：
- "如果用户输入是 X，专家会怎么回应？"
- "这种情况下，专家会停下来澄清什么？"
- "agent 如果走偏了，专家会怎么纠正？"

**SKILL.md 写法**：

```markdown
### Coaching 检查点（执行中）

| 触发信号 | 专家反馈 |
|---------|---------|
| agent 一次 Read 超过 5 个文件 | "⚠️ 等等，先 grep 定位关键行，避免上下文爆炸" |
| agent 想跳过 lint | "⚠️ 即使改动很小也要跑 lint，防止 typo" |
| agent 直接修 bug 没问根因 | "⚠️ 先 reproduce + minimisation，别盲修" |
```

### 3. Scaffolding（搭脚手架）

**核心**：执行前给用户反问的问题，让流程更稳。

**问题模板**：
- "执行前必须问用户的 1-2 个问题？"
- "用户的什么回答会改变流程？"
- "新手需要详细脚手架，老手可以省略什么？"

**SKILL.md 写法**：

```markdown
### 执行前脚手架（必问）

| 问题 | 用户回答影响 |
|------|------------|
| "这次任务是新增还是重构？" | 重构 → 启用保护测试；新增 → 委托 tdd |
| "改动范围单文件还是跨模块？" | 跨模块 → 升级复杂度为完整路径 |
| "项目用了哪些自动生成工具？"（gf gen / openapi codegen / protoc） | 命中 → 禁止手动编辑生成目录 |
```

### 4. Fading（逐步撤支撑）

**核心**：识别熟练度信号，省略冗余步骤。

**问题模板**：
- "什么信号表示用户已经熟练？"
- "什么场景下脚手架可以省略？"
- "老手 vs 新手的流程差异？"

**SKILL.md 写法**：

```markdown
### Fading 信号（按熟练度调整）

| 用户信号 | 流程调整 |
|---------|---------|
| 用户明确给改动文件路径 + 行号 | 跳过 Step 1 语义导航 |
| 用户用术语（"重构" / "TDD" / "regression"） | 跳过复杂度评估，按用户标签走 |
| 用户拒绝脚手架问题（"直接做"） | 切换快速路径，事后审查 |
| 用户首次使用本 skill | 启用全部脚手架 + Coaching |
```

## 四要素协同

**不能只用一个**。常见错误：

| 错误 | 后果 |
|------|------|
| 只用 Modeling | SKILL.md 全是专家视角，但没有用户互动节点 |
| 只用 Scaffolding | SKILL.md 全是问题清单，但没说专家怎么判断 |
| 只用 Coaching | SKILL.md 全是反馈规则，但用户不知道何时触发 |
| 只用 Fading | SKILL.md 全是熟练度分级，但新手阶段缺指导 |

**正确组合**：4 要素都写，但按 skill 类型调整权重。

| skill 类型 | 权重建议 |
|-----------|---------|
| 执行型（coder） | Modeling 40% + Coaching 30% + Scaffolding 20% + Fading 10% |
| 研究型（skill-search） | Modeling 50% + Scaffolding 30% + Coaching 15% + Fading 5% |
| 文档型（citation-sourcing） | Modeling 60% + Scaffolding 30% + Coaching 10% + Fading 0% |

---

## 与 Anthropic 反推法的关系

Anthropic 反推法（P2）告诉我们**缺什么**（失败 case 萃取）。
Cognitive Apprenticeship（P3-P4）告诉我们**怎么写**（四要素格式）。

两者必须配合使用。

## 反例 vs 正例

### 反例（流程化，无四要素）

```
3. 加载历史经验
4. 执行任务
5. 验证
6. 总结
```

### 正例（四要素齐全）

```
3. 加载历史经验

   > 💡 专家视角（Modeling）：limit=5 不是随便选的... [ expert reasoning ]

   ### 执行前脚手架（Scaffolding）
   | 问题 | 影响 |
   | "任务是新增还是重构？" | 重构 → 保护测试 |

4. 执行任务

   ### Coaching 检查点（执行中）
   | 信号 | 反馈 |
   | "Read >5 文件" | "⚠️ 先 grep 定位" |

5. 验证

6. 总结

   ### Fading 信号（按熟练度）
   | 信号 | 调整 |
   | "用户给明确行号" | 跳过 Step 1 |
```
