# Fin Code Review — 金融级代码审查

面向 Go / Python 金融业务代码的结构化审查。三大维度优先级：**数据 > 流程 > 安全**。

## 适用范围

- 语言：Go、Python
- 场景：支付、结算、风控、对账、清算、账务、定价、交易、合规
- 输入：单个功能 PR / 多功能批量 PR / 整个模块审查
- 不适用于：前端 UI、非金融后端逻辑、DevOps/infra 代码

## 审查流程

```
P0 定位 → P1 数据完整性 → P2 流程正确性 → P3 安全合规 → P4 输出
```

每步必须完成才能进入下一步。发现 Critical 级别问题时，立即输出并标记 `STOP`，建议修复后再继续。

### P0: 定位

10 秒内判断本次审查的范围和类型：

1. 读 diff 或目标文件，识别涉及的**业务模块**（支付/结算/风控/对账/账务/定价/交易/合规）
2. 判断审查粒度：单功能 / 多功能 / 整模块
3. 识别关键数据类型：金额 / 利率 / 汇率 / 余额 / 头寸 / 持仓 / 费率
4. 检查是否有对应的测试文件，标记"有测试/无测试"

### P1: 数据完整性（权重最高）

| 检查项 | Go 特征 | Python 特征 |
|--------|---------|-------------|
| **金额精度** | `float64` 用于金额 → ❌；应为 `decimal.Decimal` 或整数分/厘表示 | `float` 用于金额 → ❌；应为 `Decimal` 或整数 |
| **舍入规则** | 无显式舍入模式（四舍五入/向下/银行家舍入）→ 告警 | 同左；`round()` 默认银行家舍入，是否与业务规则一致？ |
| **溢出风险** | `int32` 存储金额 → 告警；大额场景应用 `int64` 或 `big.Int` | Python int 无溢出，但序列化时 `json` 可能丢精度 |
| **并发竞态** | `goroutine` 操作共享金额变量无 `mutex` / 无 `atomic` → Critical | 多线程/协程操作共享余额无锁 → Critical |
| **空值处理** | `nil` 指针解引用金额字段 → Critical | `None` 参与算术运算 → Critical |
| **序列化/反序列化** | JSON `float` 金额字段 → 告警；应使用字符串或整数 | 同左；`dataclass` 金额字段类型检查 |
| **边界值** | 金额 ≤ 0 的合法性检查；上限检查 | 同左 |
| **币种/单位** | 无币种标注或混用不同单位（元/分/厘）→ 告警 | 同左 |

**Go 专项**：
- `database/sql` 的 `Scan` 到 `float64` → 告警
- `strconv.FormatFloat` 金额格式化 → 检查精度参数
- protobuf 金额字段用 `double` → 告警，应使用 `string` 或 `int64`

**Python 专项**：
- `pandas` 的 `float64` 列存金额 → 告警
- Django `FloatField` 存金额 → 告警，应用 `DecimalField`
- SQLAlchemy `Float` 列存金额 → 告警

### P2: 流程正确性

| 检查项 | 说明 |
|--------|------|
| **幂等性** | 重复请求是否产生相同结果？重试、超时重发、消息重复消费是否安全？ |
| **事务完整性** | 多表/多服务操作是否在事务内？失败是否回滚？分布式事务是否有补偿？ |
| **状态机正确性** | 订单/交易状态流转是否合法？是否有非法状态跳转？ |
| **对账逻辑** | 是否有对账机制？差异处理流程是否完整？ |
| **审计日志** | 关键操作是否记录审计日志？日志是否包含操作人、时间、金额、前后值？ |
| **T+0/T+1 处理** | 是否正确处理交易日和结算日的分离？时区处理是否正确？ |
| **并发控制** | 同一账户/订单并发操作是否有锁机制？乐观锁版本号是否检查？ |
| **错误处理** | 金融操作失败是否有明确错误码？是否有兜底处理？错误是否可追溯？ |
| **超时与重试** | 外部调用超时是否合理？重试是否有退避？重试是否幂等？ |

**Go 专项**：
- `context.WithTimeout` 超时是否覆盖完整调用链
- `errgroup` 中 goroutine 失败是否正确取消其他
- database/sql 事务 `Begin` 后是否保证 `Rollback`（defer 或 error path）

**Python 专项**：
- Celery 任务重试配置是否合理（`autoretry_for` / `retry_backoff`）
- Django `transaction.atomic` 嵌套使用是否正确
- `@retry` 装饰器是否覆盖关键外部调用

### P3: 安全合规

| 检查项 | 说明 |
|--------|------|
| **SQL 注入** | 动态拼接 SQL → Critical；参数化查询是否完整？ |
| **加密传输** | 敏感数据传输是否 TLS？内部服务调用是否加密？ |
| **数据加密存储** | PII（姓名/身份证/银行卡）是否加密存储？密钥管理是否安全？ |
| **日志脱敏** | 日志中是否泄露金额/卡号/身份证号？ |
| **鉴权授权** | API 是否有鉴权？权限是否按角色区分？越权风险？ |
| **PII 处理** | 个人信息收集是否有合规依据？是否最小化采集？ |
| **合规标注** | 关键业务逻辑是否标注了适用的法规条款（如反洗钱、KYC）？ |
| **密钥硬编码** | API key / secret / token 硬编码 → Critical |
| **输入校验** | 外部输入是否有白名单校验？金额输入是否有上下限？ |

### P4: 输出格式

```markdown
## 金融代码审查报告

**审查范围**：{业务模块} | **语言**：{Go/Python} | **文件数**：X | **审查类型**：{单功能/多功能/模块}

### Critical（必须修复才能上线）
- [P1-数据] {文件}:{行号} — {问题描述} — {修复建议}
- [P2-流程] ...

### Warning（强烈建议修复）
- [P1-数据] ...
- [P3-安全] ...

### Info（改进建议）
- [P2-流程] ...

### 审查总结
| 维度 | Critical | Warning | Info |
|------|----------|---------|------|
| P1 数据完整性 | X | X | X |
| P2 流程正确性 | X | X | X |
| P3 安全合规 | X | X | X |

### 关键风险提示
{1-3 句话总结本次审查发现的最大风险}

### 测试建议
{针对发现的问题，建议补充的测试用例}
```

## 严重性定义

| 级别 | 定义 | 举例 |
|------|------|------|
| **Critical** | 可能导致资金损失、数据不一致、安全漏洞 | float64 存金额、事务不完整、SQL 注入、密钥硬编码 |
| **Warning** | 可能导致业务异常或合规风险 | 无幂等、无审计日志、日志脱敏不完整 |
| **Info** | 可改进但不影响正确性 | 命名不规范、缺少注释、可简化的逻辑 |

## 约束

1. **P1 优先** — 数据完整性问题必须最先审查，金融场景下数据错误 = 资金风险
2. **Critical 立即报告** — 发现 Critical 级别问题时在当前步骤就输出，不要等全部审查完
3. **金额相关一律不放过** — 任何涉及金额的代码（赋值/运算/比较/传递/序列化）必须逐行检查
4. **不审查非金融逻辑** — 纯 UI、路由、配置、infra 代码不在本 Skill 范围
5. **不猜测业务规则** — 如果审查时发现疑似业务规则问题（如利率计算逻辑），标记为 Warning 并建议确认，不自行判断对错
6. **Go/Python 双语检查** — 根据检测到的语言自动切换检查规则，Go 项目用 Go 专项，Python 项目用 Python 专项
