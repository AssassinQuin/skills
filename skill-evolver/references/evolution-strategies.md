# S1-S6 进化策略详细指南

Phase 2 多策略探索时读取本文件。

---

## S1: 指令精化（Instruction Refinement）

将模糊指令替换为精确操作。

### 诊断信号
- 5 维 rubric 维度4（指令精度）< 6
- 用户反馈"不知道怎么操作"
- 测试输出偏离预期

### 精化模式

| 模糊 | 精化 |
|------|------|
| "分析代码" | "1. 读取文件 2. 识别函数边界 3. 计算圈复杂度 4. 标记 >10 的函数" |
| "生成报告" | "输出 Markdown，包含：标题(##)、数据表(\|...\|)、建议列表(- )" |
| "选择合适的方法" | "按优先级：① 方法A（条件X时）② 方法B（条件Y时）③ 默认方法C" |
| "处理图片" | "1. 检测格式 2. 读取尺寸 3. 按 max(宽,高)=1024 缩放 4. 写入输出路径" |
| "优化性能" | "1. time 测量耗时 2. profile 热点 3. 针对 top-2 热点优化" |

### 精化检查清单
- 每个动词是否可被直接执行？
- 输入/输出格式是否明确？
- 参数是否有默认值或取值范围？
- 是否有可执行示例？

## S2: 工作流重组（Workflow Reorganization）

### 诊断信号
- "步骤太多" → 合并
- "不知道在哪" → 添加进度标记
- 失败集中某步 → 拆分
- 跳跃式使用 → 标记 [可选]

### 重组模式

**合并** — 连续3+小步骤合为一个阶段：
```
Before:  1.检查文件 2.读取内容 3.解析格式 4.验证完整性
After:   1. 加载并验证文件（检查→读取→解析→校验）
```

**拆分** — 过大步骤拆为带检查点的子步骤：
```
Before:  3. 生成输出
After:   3. 生成初稿 → [检查点：展示初稿确认方向]
         4. 根据反馈精化
         5. 格式化最终输出
```

**重排** — 高频步骤前置：
```
Before:  1. 完整环境检查 2. 读取配置 3. 执行核心任务
After:   1. 执行核心任务（快速路径，失败时再检查环境）
         2. [可选] 环境检查
```

## S3: 边界增强（Boundary Hardening）

### 诊断信号
- traces.jsonl 中 partial 率 > 30%
- 5 维 rubric 维度3（边界/安全）< 6
- 用户反复调整同一类输出

### 增强模式

**IF-ELSE 链** — 每步操作后加 fallback：
```
3. 执行分析
   - 数据 < 100 行 → 全量分析
   - 数据 ≥ 100 行 → 采样分析（采样率 10%）
   - 采样置信度 < 0.8 → 自动提升至 30%
```

**输入校验** — 开头加前置条件：
```
## 前置检查
- 文件格式为 .xlsx 或 .csv（否则提示转换）
- 数据行数 > 0（否则报错退出）
- 必需列 [A, B, C] 存在（否则提示缺失列）
```

**输出兜底** — 关键输出加最低保障：
```
## 输出保障
无论分析结果如何，输出必须包含：
- 结论摘要（1-2 句）
- 关键数据点（≥3 个）
- 建议列表（≥1 条）
```

## S4: 上下文优化（Context Optimization）

### 诊断信号
- 输出冗余 → skill 说明过多
- 输出不足 → 缺关键信息
- 风格不匹配 → 输出模板不对

### 优化模式

**信息过密 → 拆分**：
```
# 改前：堆在 SKILL.md
## 详细 API 参考 [500行]

# 改后：拆为 references
## API 参考
详见 references/api.md。常用：create()、query()
```

**缺少示例 → 添加**：
```
# 改前
输出 JSON 格式。

# 改后
输出 JSON：
{"summary": "...", "metrics": {"accuracy": 0.95}, "anomalies": []}
```

**风格不对 → 调整模板**：
```
# 改前
详细分析报告如下...

# 改后（严格结构）
1. 一句话结论
2. 关键发现（bullet，≤5 条）
3. 数据支撑（表格）
4. 建议行动（按优先级）
```

## S5: 范式转换（Paradigm Shift）

改变 skill 的组织范式，不只是在同一范式内微调。

### 诊断信号
- S1-S4 连续多轮都涨不动
- skill 步骤 > 15 且仍有遗漏
- 同一场景被不同路径重复处理
- 用户反馈"流程太死板/不够灵活"

### 可选范式

**线性流程 → 决策树**
```
# 改前：线性
1. 检测文件格式
2. 如果是 PDF → 步骤A
3. 如果是 DOCX → 步骤B
4. 如果是 XLSX → 步骤C
...

# 改后：决策树
## 格式 → 处理路径
- PDF → 见 Phase A（text/image 检测 → 选择策略）
- DOCX → 见 Phase B（tracked changes 检测 → 选择策略）
- XLSX → 见 Phase C（formula 检测 → 选择策略）
```

**线性流程 → 状态机**
```
# 适用场景：步骤间有复杂的前后依赖和回退
状态: IDLE → ANALYZING → PROCESSING → REVIEWING → DONE
事件: start → analyze_complete → process_complete → review_pass/reject
回退: REVIEWING --reject--> ANALYZING
```

**段落式 → 表格驱动**
```
# 改前：每个场景写一段说明
处理场景A时...
处理场景B时...
处理场景C时...

# 改后：一张决策表
| 条件 | 操作 | 输出格式 |
|------|------|---------|
| 场景A | 操作序列A | 格式X |
| 场景B | 操作序列B | 格式Y |
```

### 风险
- 范式转换是激进操作，必须征得用户同意
- 转换后需要完整 Phase 4 审计（尤其检查 Coverage 和 Silent-bypass）
- 体积可能暂时超 120%，允许到 150%（因为是结构性重组）

## S6: 拆分/合并（Skill Boundary Recombination）

改变 skill 的边界，拆分过大或合并过碎的 skill。

### 诊断信号
- SKILL.md > 500 行且有多个独立功能 → 拆分
- 多个 skill 经常被连续调用且有强依赖 → 合并
- frontmatter description 列了 > 5 个不相关的触发场景 → 拆分
- 用户经常需要"先 A 再 B"两个 skill 才能完成一个任务 → 合并

### 拆分模式

```
# 改前：一个 monster skill
pdf-editor/
├── SKILL.md (800行，含提取/创建/合并/拆分/表单)

# 改后：按功能域拆分
pdf-extract/SKILL.md   ← 提取文本/表格/图片
pdf-create/SKILL.md    ← 创建新 PDF
pdf-merge-split/SKILL.md ← 合并/拆分 PDF
pdf-forms/SKILL.md     ← 表单处理

# 每个 skill 的 description 中加互引：
# "对于 PDF 创建，使用 pdf-create skill"
```

### 合并模式

```
# 改前：两个强依赖 skill
api-design/SKILL.md    ← 设计 API
api-doc-gen/SKILL.md   ← 根据 API 设计生成文档

# 改后：合并为一个工作流
api-design-to-docs/SKILL.md  ← 设计→文档一步到位
# 保留原始 skill 的 description 触发词
```

### 约束
- 拆分后每个子 skill 的 description 必须包含原始触发词的子集
- 合并后的 description 必须包含所有原始 skill 的触发词
- 拆分/合并需要更新其他 skill 中对目标 skill 的引用
- 必须征得用户同意

## 对比学习详细操作

Phase 2 第 3 步的 Δr = ϕ(τ+) \ ϕ(τ-) 具体执行：

### 步骤 1: 逐段 Diff

```
对每个候选版本 Sk (k=1..6) 与原始版本 S0 做 diff：

S0: [段落A] [段落B] [段落C] [段落D] [段落E]
S1: [段落A] [段落B'] [段落C] [段落D'] [段落E]   ← B和D被修改
S3: [段落A] [段落B] [段落C'] [段落D] [段落E']   ← C和E被修改

差异段集合：
D(S1) = {B', D'}
D(S3) = {C', E'}
```

### 步骤 2: 分类标记

```
对每个差异段，标记其在测试中的表现：

B' → 出现在 S1 中 → S1 的测试结果 τ(S1) = success → 标记 +
D' → 出现在 S1 中 → S1 的测试结果 τ(S1) = success → 标记 +
C' → 出现在 S3 中 → S3 的测试结果 τ(S3) = failure → 标记 -
E' → 出现在 S3 中 → S3 的测试结果 τ(S3) = failure → 标记 -

如果同一差异段出现在多个候选中：
B' → S1(success) 和 S2(success) → 都标 +
D' → S1(success) 和 S4(failure) → 标记 =（中性，不是差异因素）
```

### 步骤 3: 提取 Δr

```
Δr = {+标记的段} \ {=标记的段}
    = {B'} \ {}  （排除中性段 D'）
    = {B'}

B' 是成功的关键差异。
```

### 步骤 4: 排序

```
差异段在成功版本中出现的频率：
B': 3/4 个成功版本都有 → 高置信度
D': 1/4 → 低置信度

按频率降序排列 → B' 优先作为补丁候选。
```

### 输出格式

```
Δr = [
  {段: B', 频率: 3/4, 策略: S1, 置信度: 高},
  {段: D', 频率: 1/4, 策略: S1, 置信度: 低},
]
→ Phase 3 外科补丁优先应用 B'
```
