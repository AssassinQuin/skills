# Simplify Guide

Fix 模式的代码简化原则。核心：保留行为，提升可读性。

## 五原则

1. **保留行为** — 输入/输出/副作用/错误/边界全部不变。不确定就不改。
2. **遵循项目惯例** — 先读项目规范和相邻代码风格。打破一致性的"简化"是折腾。
3. **清晰优于紧凑** — 嵌套三元→可读控制流；密集内联→命名中间步骤。
4. **不过度简化** — 不内联有意义的名称、不合并不相关逻辑、不删除服务可测试性的抽象。
5. **范围限于变更** — 默认只简化最近修改的代码，不做无关重构。

## 流程

1. **先理解** — 代码职责？调用方？边界？测试？为什么这么写？答不出就读更多。
2. **找信号** — 深嵌套、长函数、嵌套三元、布尔参数、重复条件、模糊命名、死代码、无用包装。
3. **逐项应用** — 每次一个简化→运行测试→保留仅当行为不变。
4. **验证** — 更易懂？diff 干净？惯例匹配？无副作用变化？

## 适用场景

- 功能完成且测试通过，但实现偏重
- review 中标记了可读性/复杂度问题
- 深嵌套、长函数、不清晰命名
- 时间压力下的代码需要清理

## 不适用

- 代码已经干净
- 还不理解代码行为
- 性能关键且"简化版"更慢
- 即将整体重写
