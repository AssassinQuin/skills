# Code Review — 代码审查

审查 git diff 变更，输出变更理解 + 结构化审计报告 + 修复建议。

## 模式判定

| 输入 | 模式 | 流程 |
|------|------|------|
| "快速审查" / ≤ 3 文件简单变更 | Quick Scan | P0 → P1 → P3 |
| PR / diff / 指定文件 | Deep Review | P0 → P1 → P2 → P3 → P4 |
| > 10 文件或 > 500 行 | Batch | 分批 ≤ 10 文件，逐批 Deep Review |

## 执行流程

```
P0 定位 → P1 理解变更 → P2 深度审查 → P3 交叉验证 → P4 输出报告
```

### P0: 定位

1. 检测语言（go.mod / pyproject.toml / package.json / Cargo.toml）
2. 获取 diff 范围（`git diff` / 指定文件 / PR）
3. 复杂度：简单(≤ 3 文件, ≤ 200 行) / 标准 / 复杂(≥ 10 文件或涉及并发/API/金额)

### P1: 理解变更

**推断作者意图**：分析 diff 整体模式，推断最可能的意图。例如：
- "意图：重构 `calculate_total` 提升可读性"
- "意图：修复分页逻辑的 off-by-one 错误"
- "意图：为 `process_user` 添加 null 检查防止 NPE"

**收集关键上下文**：必须调用工具读取相关源文件和调用方，不能凭 diff 猜测。

**输出变更摘要**：
- 涉及的模块/功能域
- 变更类型（新功能/修复/重构/性能优化/安全补丁）
- 关键数据流变化
- 受影响的接口/调用链

**可视化变更流程**（Deep/Batch 模式必须，Quick Scan 可选）：

用 Mermaid 图示变更的核心流程。简单变更 1 张图，复杂变更（跨模块/多方面）2 张：

- **业务流**：用户操作/数据流向
- **技术流**：调用链/请求处理

要求：
- 优先用 flowchart/sequenceDiagram 展示逻辑链和调用关系
- 不用静态分类盒图
- 用颜色高亮变更部分
- 同时指定 fill 和 color 确保明暗主题可读（如 `fill:#c8e6c9,color:#1a5e20`）

### P2: 深度审查

7 维逐项审查，以 P1 推断的意图为上下文：

**D1 正确性**：边界条件、错误处理完整性、逻辑分支、类型安全

**D2 架构(SOLID)**：函数过长→违反 S；switch 散布→违反 O；接口过大→违反 I

**D3 安全**：注入(SQL/XSS/命令)、认证授权、敏感数据、加密

**D4 性能**：算法复杂度、N+1 查询、内存、IO

**D5 可读性**：命名、函数长度(≤50 行)、嵌套(≤3 层)

**D6 测试**：核心路径覆盖(正常+边界+错误)

**D7 冗余代码**：注释代码/未使用/重复(>80%)/过度抽象/过时 TODO

**敏感场景自动提升**：涉及金额/并发/敏感数据时，D1+D3 检查深度自动提升。

### P3: 交叉验证

**派 2 个子 agent 并行验证所有发现**：
- 每个 agent 独立读取相关代码上下文
- 逐项检查：问题是否真实存在？行号是否正确？严重性是否合理？
- 置信度评分：

| 验证结果 | 置信度 | 处理 |
|----------|--------|------|
| 2/2 确认 | High | 纳入报告 |
| 1/2 确认 | Medium | 纳入报告但标注 |
| 0/2 确认 | Low | 排除 |

子 agent 不可用时的降级：自行重读每个发现的代码上下文，逐条挑战"这确定是问题吗"，无法确认的排除。

### P4: 输出审计报告

无发现时：简短说明代码质量良好，附 P1 变更摘要和 Mermaid 图。

有发现时，完整输出：

```markdown

**范围**：{N} 文件 {M} 行 | **语言**：{lang} | **模式**：{Quick/Deep/Batch}

### 变更摘要
- **涉及模块**：{模块列表}
```

## Agent 消费接口

报告中的 `优先级` 和 `置信度` 供其他 agent 判断是否修复：

| 优先级 | 含义 | 自动修复建议 |
|--------|------|-------------|
| `MUST-FIX` | 安全漏洞、数据丢失、必现 bug | 建议立即修复 |
| `SHOULD-FIX` | 高概率 bug、架构问题 | 建议本轮修复 |
| `NICE-TO-FIX` | 边界、可维护性 | 可选 |
| `OPTIONAL` | 风格、命名 | 仅明确需求时处理 |

## 约束

1. **Review-only** — 只输出发现，不修改代码（除非用户明确要求修复）
2. **先理解后审查** — 必须先 P1 理解意图，再 P2 审查问题
3. **P0 立即报告** — 发现 P0 在当前步骤就输出
4. **证据驱动** — 每个发现必须有 `{文件}:{行号}`，无证据不输出
5. **交叉验证** — Deep/Batch 模式必须经过 P3 交叉验证才输出
6. **禁止低价值评论** — 不做纯描述性、赞美性、无行动建议的评论；不用"可能""也许""你想看看"
7. **跳过非代码文件** — 不审查 .md/.json/.txt/.svg/.lock
8. **编译语言假设** — 静态类型语言假设编译通过，不报告类型错误（除非 diff 证明有问题）
9. **尊重有意删除** — 大段功能删除视为有意为之
10. **去重合并** — 相似问题合并为一个，跨文件重复给一个代表+列出其他位置
11. **语言一致** — 输出语言跟随用户输入语言（中文提问→中文报告）
