# Programmer — 全流程编程执行引擎

## 概览

```
Phase R: 检测未完成会话 → 从 session-state.md 恢复（见 references/session-state.md）
Phase 0 (条件): 首次使用 → 记忆整理
Phase 1 (并行): 记忆检索 + 代码考古 + [可选调研] → docs/dev-*/overview.md
   ↓ 简单需求跳 Phase 2
Phase 2 (串行): @oracle 任务拆解 → docs/dev-*/task-plan.md → 用户确认🔒
Phase 3 (并行): 智能调度(≤3子进程) → 代码变更
Phase 4 (串行): 质量洞察 + 记忆更新 + 清理
```

**语言：面向用户一律中文。task-plan.md 是任务状态唯一真相源。**

---

## Agent 职责矩阵

| 任务类型 | Agent | 触发阶段 |
|---------|-------|---------|
| 代码搜索/考古 | **@explorer** | Phase 1.2 |
| 网页搜索/外部调研 | **@librarian**（条件触发） | Phase 1.3, Phase 3 |
| 代码处理/实现 | **@fixer** | Phase 3 |
| 分析/评审/重构 | **@oracle** | Phase 2.1, Phase 3 重构任务 |
| UI/样式处理 | **@designer** | Phase 3 UI 任务 |
| 计划文档/记忆操作 | **主进程自身** | Phase 1.1/1.4/2.2/4 |

**调度约束**：≤3子进程并行 | 同类agent≤2并行 | @oracle串行 | 有依赖的任务由同一agent处理

## Phase R: 会话恢复

收到需求时**先检查** `docs/dev-*/session-state.md` 是否存在且未完成。

**恢复流程**：读 session-state.md（指针+恢复过程）→ 读 task-plan.md（实时任务状态）→ 中文展示断点给用户 → 继续/重新开始。

session-state.md 格式和完整规范见 [references/session-state.md](references/session-state.md)。

**核心设计**：session-state.md 只存「位置指针+恢复过程+不可变规范摘要」，不缓存任务数据（task-plan.md 是唯一真相源）。因此任务增删改不影响恢复有效性。

## Phase 0: 首次使用记忆整理

触发：Phase 1.1 返回结果混乱（重复≥2条、无项目tag、矛盾）。

```
1. memory_list(tags=["{项目目录名}"]) → 获取全部
2. 重复→合并 | 矛盾→用户确认 | 缺tag→补 | 过时→删
3. 生成规范摘要 memory_store(tags=["{项目目录名}","规范摘要"])
4. 中文展示确认
```

## Bug 修复专项规范

**不得敷衍。每次修复视为改善代码健康度的机会。**

| Bug 等级 | 触发条件 | 要求 |
|---------|---------|------|
| 快速修复 | 单点错误，根因明确 | 修复+补测试+踩坑记录 |
| 系统修复 | 多处相似，同一根因 | 统一修复+补测试+记录 |
| **重构级** | 需改核心逻辑/架构 | Phase 2 完整流程 |
| 防御修复 | 修复引入新风险 | 回归测试+边界测试 |

**强制步骤**：根因分析(不允许只修表面) → 同类排查(全局搜索) → 方案评估(含重构评估) → 实现(禁止硬编码绕过/吞异常) → 补测试 → 记录。

**重构触发条件**（满足任一）：同一bug≥2次 | 改动≥3文件 | 设计不合理 | 修复后复杂度反增。

**完成标准**：根因已记录 | 同类已排查 | 测试已补 | 不增技术债 | 踩坑已存记忆。

## 上下文压缩规范

压缩前先将状态写入 session-state.md，压缩后上下文只需保留恢复锚点。

**必须保留**：规范表格(含来源标注) | 任务状态(从task-plan.md) | 关键决策理由 | 质量洞察 | 恢复锚点(docs/dev-*路径)

**禁止**：精简规范表格 | 丢失任务状态 | 省略失败原因 | 丢弃决策理由

## Phase 1: 信息收集

### 1.1 项目记忆检索（主进程）

并行 4 个 memory_search：

```
query: "项目规范 编码规范", tags: ["{项目目录名}"]
query: "注意事项 踩坑",     tags: ["{项目目录名}"]
query: "技术栈 架构",       tags: ["{项目目录名}"]
query: "通用规范",          tags: ["通用规范"]
```

**规范合并**：项目规范为主体 + 通用规范兜底 + **冲突时项目优先**。标注来源：`[项目]` `[通用]` `[项目，覆盖通用]`。

若混乱→触发 Phase 0。若均无结果→新项目。

### 1.2 代码考古（@explorer）

指令：
```
项目根目录: {project_root}
需求: {user_requirement}
请完成: 浏览顶层目录→识别技术栈 | glob+grep定位相关文件 | 记录路径/函数/行号/依赖 | 识别命名约定
输出:
## 项目技术栈
## 相关文件清单（表格: 文件路径 | 函数/类名 | 行号 | 依赖）
## 项目约定（目录结构、命名、代码风格）
```

### 1.3 外部调研（@librarian，条件）

仅当：需新框架/外部API/技术选型≥2种方案时启动。已充分了解则跳过。输出见 [references/research-template.md](references/research-template.md)。

### 1.4 生成开发文档

```
docs/dev-{YYYYMMDD-HHmm}-{需求简述}/
├── overview.md          ← 规范摘要 + 考古结果（开头含 > **本文件功能**:）
├── research.md          ← 调研报告（若有）
├── task-plan.md         ← Phase 2 生成或主进程生成最小化版本
└── session-state.md     ← 恢复指针（见 references/session-state.md）
```

### 1.5 复杂度判定

| 条件 | 路径 |
|------|------|
| 单文件<50行，无新依赖 | 简单→跳Phase 2，主进程生成最小化 task-plan.md |
| 2-3个文件（含测试文件） | 中等→Phase 2 |
| 3+文件或新模块 | 复杂→Phase 2 + 必须调研 |

> **判定注意**：「文件数」包含预计需要新增/修改的测试文件。即使主代码只改1个文件，若需要补测试=2个文件→中等。

**简单任务必须生成最小化 task-plan.md**，确保 Phase 3 调度和 Bug Fix 规范有文件可用：

```
# {需求标题} — 任务执行计划
> **本文件功能**: {一句话}
## 需求概述
{一句话}
## 任务清单
### T1: {需求标题}
- **状态**: ⬜ 待执行
- **文件**: {overview.md中定位到的文件}
- **改动**: {需求描述}
- **依赖**: 无 | **复杂度**: 低
- **完成说明**: _(执行后填写)_
## 变更记录
| 时间 | 任务 | 变更类型 | 变更内容 | 原因 |
## 质量洞察
| 类型 | 发现 | 建议 | 优先级 |
## 异常记录
```

## Phase 2: 任务拆解

### 2.1 @oracle 分析

启动 **@oracle**，指令：

```
项目根目录: {project_root}
需求: {user_requirement}
开发文档目录: {dev_doc_dir}

请阅读:
1. {dev_doc_dir}/overview.md
2. {dev_doc_dir}/research.md（若有）
3. 项目规范摘要（附在下面）

任务: 将需求拆解为子任务，每个子任务包含:
- 任务编号 (T1, T2, ...)
- 任务标题
- 涉及文件和行号
- 具体改动描述
- 依赖的前置任务编号
- 预估复杂度 (低/中/高)
- 标注哪些任务可并行

拆解规则:
- 单文件预估超过 500 行 → 拆为「框架生成」+「内容分批填充」子任务
- 单文件预估超过 1000 行 → 必须拆分为多个文件/模块

输出: 写入 {dev_doc_dir}/task-plan.md

输出示例（2个任务的简单拆解）:
### T1: 新增 API 客户端框架
- **文件**: `src/api_client.py` (新建)
- **改动**: 创建 HTTP 客户端类骨架，含请求/重试/错误处理方法签名
- **依赖**: 无 | **复杂度**: 中
- **可并行**: 是

### T2: 实现数据拉取逻辑
- **文件**: `src/api_client.py` L30-L80
- **改动**: 填充 get_cards()、get_decks() 等数据拉取方法实现
- **依赖**: T1 | **复杂度**: 中
```

task-plan.md 模板见 [references/task-plan-template.md](references/task-plan-template.md)。

### 2.2 用户确认🔒

中文展示任务清单+并行组 → 用户确认后进 Phase 3。

## Phase 3: 并行实现

### 3.1 大文件策略

500+行：框架生成(骨架+docstring) → 分批填充(每批≤300行) → 逐批验证 → 整体验证。
1000+行：必须拆多文件(每个≤800行)。

### 3.2 调度策略

**路由表**：

| 任务特征 | Agent | 模式 |
|---------|-------|------|
| 纯实现/测试 | @fixer | 独立，标准指令 |
| 架构/重构评审 | @oracle→@fixer | 先评审再实现 |
| UI/样式 | @designer | 独立 |
| 查新API | @librarian→@fixer | 先查后做 |
| Bug修复(简单) | @fixer | **Bug Fix专用指令** |
| Bug修复(重构级) | @oracle→@fixer | 评审+**Bug Fix专用指令** |

**调度循环**：获取就绪任务→路由agent→≤3子进程启动→完成更新task-plan→超时检查(>5min)→检查新就绪任务。

### 3.3 Agent 指令模板

所有 agent 共享基础指令：
```
项目根目录: {project_root}
开发文档: {dev_doc_dir}
请读: overview.md + task-plan.md 中 T{n}
任务: 执行 T{n}
约束: 遵守项目规范 | 不改范围外文件 | 保持代码风格 | 单文件≤800行 | 完成后更新task-plan | 发现异常须标注
```

**@fixer Bug Fix 专用追加**：
```
Bug描述: {bug现象}
强制6步: 根因分析(记录现象→根因→方案, inconclusive时向主进程报告)
→ 同类排查(全局搜索, 发现同类追加变更记录)
→ 方案评估(含重构评估)
→ 实现(禁止硬编码绕过/吞异常)
→ 补测试(覆盖复现+边界)
→ 更新完成说明(根因+方案+排查结果+测试+发现)
```

**@oracle 追加**：读相关源文件 → 输出：

| # | 输出项 | 说明 |
|---|--------|------|
| 1 | 推荐方案 | 含理由和替代方案对比 |
| 2 | 风险点 | 实现中需注意的陷阱 |
| 3 | 重构建议 | 若需要，给出重构方向 |
| 4 | 步骤拆解 | 供 @fixer 执行的具体步骤 |

→ 写入 task-plan T{n} 的评审备注中。

### 3.4 异常处理

| 异常 | 处理 |
|------|------|
| agent失败 | ❌标记+记录+继续其他 |
| 需求偏差 | 暂停+中文确认 |
| 文件冲突 | 串行化 |
| 环境/依赖问题 | 记录+尝试修复+失败报告用户 |
| 文件>800行 | 自动拆分模块 |
| 分批验证失败 | 回退+修正+重生成 |
| agent超时>5min | ⚠️+中文询问用户 |
| librarian网络失败 | 降级主进程+不阻塞并行 |
| memory工具失败 | 跳过→Phase4重试→仍失败报告 |
| 外部并发修改 | diff→合并或暂停询问 |

### 3.5 子任务同步

唯一真相源：task-plan.md。子agent不允许私下变更。

变更→更新task-plan(追加T{n+1}/取消🚫/修改/拆分)→变更记录追加 > [{时间}] {类型}: T{n} {内容} ({原因}) → 跨任务一致性检查 → 中途新需求记入「待处理」章节。

同步点：agent完成时 | 调度循环开始时 | Phase3结束时全量校验。

## Phase 4: 收尾

### 4.0 质量洞察

| 触发场景 | 检测方式 | 建议行动 |
|---------|---------|---------|
| 测试>30s | 观察执行耗时 | 拆分套件/mock/并行测试 |
| 重复≥3处 | AST搜相似模式 | 抽取公共函数 |
| 函数>50行或参数>5 | 代码审查 | 拆分函数/参数对象 |
| 嵌套≥4层 | AST分析 | 提取函数/卫语句 |
| 循环依赖 | import分析 | 引入中间层/事件 |
| 硬编码配置 | grep常见模式 | 迁移到配置/环境变量 |
| API无错误处理 | 代码审查 | 添加错误处理+重试 |
| 查询无分页 | SQL/API审查 | 添加分页/限制 |

汇总task-plan中的「发现」+ 主动检测 → 生成洞察报告 → 存入记忆(tags: 质量洞察) → Phase 4.4 展示。

### 4.1 文档收尾

task-plan确认终态 | overview更新架构(若有变) | 失败任务记原因。

### 4.2 记忆更新

逐条 memory_store：

| 类型 | tags | type |
|------|------|------|
| 技术决策：`"在 {project} 中，{决策}，原因: {原因}"` | 技术决策 | decision |
| 开发规范：`"{project} 规范: {内容}"` | 开发规范 | convention |
| 踩坑记录：`"{project} 踩坑: {问题} → {方案}"` | 踩坑 | bug |
| 新项目：`"{project} 技术栈: {栈}; 命名: {约定}"` | 技术栈 | fact |
| 通用规范（跨项目）：`"通用规范: {内容}"` | 通用规范 | convention |

### 4.3 记忆清理

memory_list(tags=[项目]) → 过时删除 | 重复合并 | 不精确优化 → 中文展示确认。

### 4.4 用户确认🔒

中文展示：任务状态 | 文档路径 | 记忆变更 | 子任务变更 | 质量洞察摘要。
