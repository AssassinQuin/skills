# session-state.md 格式与规范

> 本文件是 programmer skill 的 references，定义 session-state.md 的完整格式。

## 设计原则

- session-state.md 是恢复的「指南针」，不是数据的「快照」
- task-plan.md 是任务状态的唯一真相源
- session-state.md 只存：位置指针 + 恢复过程 + 不可变上下文（规范摘要）
- 任务增删改不影响恢复有效性（因为不缓存任务数据）

## 格式

```markdown
# 开发会话状态

> **本文件功能**: 记录开发会话的恢复指针和不可变上下文，支持中断后恢复。
> **重要**: 任务状态以 task-plan.md 为准，本文件不缓存任务数据。

## 基本信息（创建时写入，不再变更）
- **需求**: {原始用户需求}
- **项目根目录**: {project_root}
- **开发文档目录**: docs/dev-{ts}-{name}/
- **技术栈**: {Phase 1.2 识别结果}
- **创建时间**: {ISO时间}
- **Git 分支**: {功能分支名，如 feat/20260428-xxx} 或「非 Git 项目」

## 规范摘要（Phase 1.1 结果，创建时写入）
{合并后的完整规范表格，含来源标注 [项目]/[通用]/[项目，覆盖通用]}

## 当前位置指针（每次关键步骤更新）
- **当前阶段**: Phase {0|1|2|3|4}.{子步骤}
- **最后更新**: {ISO时间}

## 恢复过程
### Phase 1 恢复
1. 检查 overview.md 和 research.md 是否已生成
2. 未完成→继续子步骤 | 已完成→Phase 1.5 复杂度判定

### Phase 3 恢复
1. 读取 task-plan.md
2. 找 ⬜ 待执行 且前置 ✅ 的任务
3. 按 Agent 职责矩阵路由表分配 agent
4. 启动 ≤3 子进程
5. 完成后更新 task-plan → 回步骤2
6. 全部完成 → Phase 4

### Phase 4 恢复
1. 读 task-plan.md 确认终态
2. 汇总质量洞察
3. 记忆更新+清理
4. 展示摘要给用户确认

## 恢复历史
| 恢复时间 | 恢复原因 | 从哪个阶段恢复 | 结果 |
|---------|---------|-------------|------|
```

## 检查点写入规则

只更新指针和过程，不缓存可变数据：

| 时机 | 写入内容 |
|------|---------|
| Phase 1.4 创建文档 | 基本信息 + 规范摘要 + Phase 1 恢复过程 |
| Phase 2 完成后 | 指针→Phase 2 + 更新恢复过程 |
| Phase 3 开始前 | 指针→Phase 3 + 更新恢复过程 |
| Phase 切换时 | 指针+恢复过程更新 |
| Phase 4 完成 | 标记 completed |

**不写入**（因为会过期）：任务状态快照 | 活跃子agent列表 | 已完成文件变更

## 中断恢复策略

| 中断类型 | 恢复策略 |
|---------|---------|
| 上下文溢出/压缩 | session-state.md 磁盘持久，读指针+过程→从task-plan读实时状态 |
| 用户 /clear | 同上 |
| 子agent超时/失败 | 检查task-plan中❌/⚠️任务，按异常处理表处理 |
| 用户中途修改需求 | task-plan变更记录跟踪，session-state恢复过程不受影响 |
| 多次中断 | 恢复历史追加，无需重复Phase 1 |
| 任务增删改 | task-plan是唯一真相源，恢复过程是通用的 |
