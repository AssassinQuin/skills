# 调研报告：MCP Docker 容器自动关闭方案

> 调研时间：2026-04-29
> 调研范围：MCP 协议规范、Docker 容器生命周期管理、进程信号处理、社区实践
> 方法：@librarian 代码考古 + Docker 官方文档检索

---

## 背景与目标

MCP（Model Context Protocol）服务器通常以 Docker 容器形式运行，通过 stdio 传输与 Agent 通信。当 Agent 关闭时，需要确保 Docker 容器自动清理，避免孤儿容器占用资源。本报告调研了所有可行的自动关闭方案，覆盖 Docker 原生机制、MCP 协议钩子、进程管理工具等。

---

## 核心发现

### 一、Docker 原生机制

#### 1.1 `--rm` 标志（⭐ 首选方案）

所有 MCP 官方参考服务器统一使用此模式 [来源 2, 6]:

```bash
docker run --rm -i mcp/server-name
```

**工作原理**：容器进程退出后，Docker daemon 自动删除容器文件系统和匿名卷。无论退出原因（正常退出、异常崩溃、信号终止）都触发清理 [来源 6]。

**MCP 场景完整流程** [来源 1, 2]:
```
Agent 关闭 → Client 关闭 stdin pipe → Server 收到 EOF/SIGTERM
→ Server 进程退出 → Docker 检测退出 → --rm 自动删除容器 ✅
```

**异常情况兜底** [来源 6]:
- Server 崩溃 → Docker 仍删除容器（`--rm` 生效）
- Server 挂死 → Client 发 SIGTERM → SIGKILL → 容器被删除
- Client 崩溃 → Docker 检测管道断裂 → 容器退出 → 被删除

#### 1.2 `--init` 标志（辅助方案）

Docker 1.13+ 内置 Tini 作为 PID 1，确保信号正确转发 [来源 9, 14]:

```bash
docker run --rm --init -i mcp/server-name
```

**解决的问题**：Shell 形式 ENTRYPOINT/CMD 会拦截 SIGTERM，导致容器无法优雅退出。`--init` 插入轻量级 init 进程，正确转发信号并回收僵尸进程 [来源 9]。

#### 1.3 PID 1 信号传播

Docker 发送 SIGTERM 到 PID 1。若使用 exec 形式 `CMD ["python", "app.py"]`，信号直达应用；shell 形式 `CMD python app.py` 会被 bash 拦截 [来源 8]。

**最佳实践** [来源 8]:
```dockerfile
# ✅ 正确 — exec 形式
ENTRYPOINT ["./myapp"]
CMD ["python", "app.py"]

# ❌ 错误 — shell 形式，信号被拦截
CMD python app.py
```

### 二、MCP 协议级钩子

#### 2.1 `onclose` 回调

MCP SDK 提供 `server.server.onclose` 在连接关闭时触发，用于应用级资源清理 [来源 3]:

```typescript
server.server.onclose = async () => {
  await closeDatabase();
  await flushBuffers();
  // 容器随后由 --rm 自动删除
};
```

**社区广泛使用**：MongoDB MCP 服务器、Dashlane CLI、Vercel MCP handler 等均实现此钩子 [来源 5]。

#### 2.2 `cleanup()` 函数模式

Everything Server 实现了工厂模式 [来源 3]:
```typescript
const { server, cleanup } = createServer();
// 断连时
cleanup(sessionId);
await server.close();
```

#### 2.3 HTTP 传输的 DELETE 端点

HTTP 长连接型 MCP 服务器支持 `DELETE /mcp` 显式终止会话 [来源 4]:
```typescript
app.delete('/mcp', async (req, res) => {
  transports.delete(sessionId);
  cleanup(sessionId);
  res.status(204).send();
});
```

### 三、兜底方案：Label + 定时清理

防止异常退出遗留孤儿容器的安全网 [来源 10]:

```bash
# 启动时标记
docker run --rm -i --label mcp-server=true --label parent-pid=$$ mcp/server

# 定时清理脚本（cron 每 5 分钟）
#!/bin/bash
docker ps --filter "label=mcp-server=true" --format "{{.ID}}\t{{.Label \"parent-pid\"}}" | \
  while read id parent_pid; do
    if ! kill -0 $parent_pid 2>/dev/null; then
      docker rm -f $id
    fi
  done
```

## 方案对比

| 方案 | 自动清理 | 自定义清理逻辑 | 实现复杂度 | MCP 实际使用 |
|------|---------|--------------|-----------|-------------|
| `--rm` | ✅ 立即 | ❌ | ⭐ 简单 | ✅ 所有官方服务器 |
| `--init` | ✅ 信号转发 | ❌ | ⭐ 简单 | ✅ 推荐 |
| MCP `onclose` | ⚠️ 应用级 | ✅ | ⭐⭐ 中等 | ✅ 广泛使用 |
| Label + cron | ⚠️ 延迟(分钟级) | ✅ | ⭐⭐⭐ | ✅ 兜底方案 |
| HEALTHCHECK | ❌ 需外部 | ⚠️ 有限 | ⭐⭐⭐ | ❌ 少用 |
| Event API | ⚠️ 需常驻进程 | ✅ | ⭐⭐⭐ | ❌ 少用 |
| systemd/supervisor | ✅ | ✅ | ⭐⭐⭐⭐ | ❌ 过重 |

## 结论与建议

### 推荐方案（三层防护）

1. **第一层（必需）**：`docker run --rm -i` — 容器退出即删除，覆盖 99% 场景 [来源 2, 6]
2. **第二层（推荐）**：`server.onclose` 钩子 — 应用级资源释放（数据库、缓冲区）[来源 3]
3. **第三层（兜底）**：Label 标记 + cron 定时清理 — 防止孤儿容器 [来源 10]

### 最佳实践模板

```bash
# 启动命令
docker run --rm --init -i \
  --label mcp-server=true \
  --label parent-pid=$$ \
  mcp/server-name
```

```typescript
// 服务器代码
server.server.onclose = async () => {
  await closeDatabase();
  await flushBuffers();
};
```

### 不推荐的方案

- ❌ Docker socket proxy（安全风险大）
- ❌ HEALTHCHECK 用于 stdio MCP（容器设计为临时性的）
- ❌ systemd/supervisor（对 MCP 场景过重）
