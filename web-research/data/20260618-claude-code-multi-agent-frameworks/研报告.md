# Claude Code 多 Agent 协作插件/框架深度调研

**日期**：2026-06-18
**触发**：用户找 Claude Code 多 agent 协作插件/框架（通用任务：研究/写作/审查/分析/工作流自动化）
**调研模式**：主 agent 直做（3 子 agent 因 sonnet API 529 全失败，主 agent MCP 正常）
**verdict**：通用任务首选 **Claude Code 原生 Subagents + Dynamic Workflows**

---

## 核心结论（先看这里）

| 场景 | 推荐 | 理由 |
|------|------|------|
| **通用任务（写作/研究/分析）** | **Claude Code 原生 Subagents + Dynamic Workflows** | 官方 + 0 第三方依赖 + 文档全 |
| **轻量多 specialist 编排** | **Maestro** | 80k token/12 分钟（vs Ruflo 220k/35 分钟）|
| **多模型互查 / 审查** | **Claude Octopus** | 8 模型并行 + 75% 共识门 |
| **CI/CD merge gate** | **Codex Peer Review** | 三 reviewer 顺序门 |
| **跨 harness / 企业** | **omnigent** | Claude/Codex/Cursor/Pi 全覆盖 + policy/sandboxing |
| **自托管 / 数据敏感** | **swarmclaw** | 自托管 + 23+ provider（含 Ollama）|
| ❌ **避开** | **Ruflo** | 100+ agents 多是注水模板 + Hive Mind 零学习效果 |

## Anthropic 官方基准（最重要）

### [How we built our multi-agent research system](https://www.anthropic.com/engineering/multi-agent-research-system)（2025-06-13）

Anthropic 自家调研系统（Claude Research 功能底层）核心结论：

1. **多 agent 主要价值是"花足够 token 解决问题"**（Anthropic 原话："Multi-agent systems work mainly because they help spend enough tokens"）
2. **必须 ≥ 200K context + Memory 持久化**（context 超 200K 会被截断）
3. **反模式**：早期错误——为简单查询 spawn 50 subagents / 无限搜不存在的源 / 互相用过多更新干扰
4. **关键判断**："你的任务是否有足够独立线程让并行性回本？"

### [Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)

先用简单 prompt + 全面评估，简单方案不够才加多步 agentic 系统。

### [Dynamic Workflows](https://code.claude.com/docs/en/workflows) + [Subagents](https://code.claude.com/docs/en/sub-agents)

Claude Code 原生支持——Claude 写脚本调度 subagents，用户可重跑。适用 codebase 审计 / 大迁移 / 交叉验证研究 / 通用任务。

## 💡 三个反常识发现

### 1. Ruflo 高 stars ≠ 高质量

Gerald Chen 实测：100+ agents 多是相似 prompt 模板；Hive Mind 跨 5 任务零学习效果；"企业级架构"是营销话术。

→ **GitHub stars / README 卖点 ≠ 实战质量**。必须看独立第三方实测。

### 2. 多 agent 主要价值是"花足够 token"（Anthropic 自承）

不是 agent 数量魔法，是 token 预算魔法。多 agent 真正优势是**并行 + 多样性 + 上下文隔离**。

### 3. 8 模型共识 ≠ 永远对（Claude Octopus 边界）

8 个模型都可能错（共同盲点）；共识可能压制正确少数；共识未达需人工介入。**共识是辅助，不是替代判断**。

## 调研执行透明度

| 维度 | 执行情况 |
|------|---------|
| spawn 子 agent 数 | 3（tech-compare + product + methodology） |
| 成功 agent | 0（**全部 API 529 sonnet 网关过载**） |
| 主 agent 补救 | git mcp 5 调用 + searxng 5 调用 + memory 1 |
| 预算超支 | 否 |
| MCP 故障 | 无（github/searxng 主 agent 用着正常） |
| 关键发现来源 | Gerald Chen 横评 + Anthropic 官方 + GitHub 搜索 |

**529 根因分析**：MCP 共享 ✓，LLM 模型独立 ❌。子 agent 用 sonnet（按 v2.3 协议），主 agent 用 glm-5.2（不同网关）。sonnet 网关过载导致 529，与 MCP 无关。

→ **暴露 v2.3 设计 gap**：缺模型降级链（sonnet → haiku → 主 agent model）。建议 v2.4 加。

## ⚠️ 知识空白

1. **中文社区独立评测少**：除 Text Matrix 的 Ruflo 指南外，缺中文独立实测
2. **长期使用反馈缺**：所有候选都 2026 H1-H2 创建，缺 6+ 月稳定性数据
3. **token 成本数据不全**：除 Gerald Chen 4 框架实测外，其他无独立成本数据
4. **通用任务专项工具稀少**：大多数框架聚焦 coding，少数（如 Academic-Writing）是个人项目
5. **失败模式文档少**：除 Anthropic 自承"早期错误"外，第三方框架失败案例少
