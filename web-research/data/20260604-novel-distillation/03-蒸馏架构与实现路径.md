# 03-蒸馏架构设计与实现路径

## 一、Grammar Builder 模式（从医疗蒸馏迁移）

Nature npj Precision Oncology 论文，TU Dresden/Heidelberg KatherLab（2025年9月），开源代码 github.com/KatherLab/LLMAIx。

### 1.1 四阶段流水线

| 阶段 | 任务 | 小说蒸馏映射 |
|------|------|------------|
| Define | 定义抽取目标schema | 定义世界观/人物/情节JSON schema |
| Preprocess | 文档预处理和分块 | 按章节分块+500词重叠 |
| Extract | Grammar Builder约束JSON输出 | 角色化Agent按schema填充 |
| Evaluate | 金标准对比评估 | 人工抽样+交叉验证 |

### 1.2 Grammar Builder 核心机制

通过 JSON schema 约束 LLM 输出格式，确保结构化一致性。例如：

```json
{
  "world_settings": {
    "power_system": { "type": "string", "description": "力量体系名称和分级" },
    "geography": { "type": "array", "items": { "type": "object" } },
    "cultural_norms": { "type": "array" },
    "historical_events": { "type": "array" }
  },
  "characters": {
    "name": { "type": "string" },
    "aliases": { "type": "array" },
    "power_level": { "type": "string" },
    "relationships": { "type": "array" },
    "arc_summary": { "type": "string" }
  }
}
```

**性能数据**：4-bit量化Llama 3.1 70B（VRAM从139GB降至43GB），在TCGA病理报告上达到87%准确率。单卡A100/H100可运行。

[来源: nature.com/articles/s41698-025-01103-4, github.com/KatherLab/LLMAIx] ✅ Nature论文+开源代码

## 二、角色化Agent分工（从法律文档迁移）

arXiv:2604.01529 (UW/GIT)：3个专业化角色分别抽取不同维度信息。

### 2.1 效果数据

| 方案 | 精确匹配率 | 提升 |
|------|-----------|------|
| 单一通用prompt（基线） | 19-27% | — |
| 3角色化agent | **44.16%** | **2-3倍** |

### 2.2 小说蒸馏的角色化架构

```
角色1: 世界观架构师 (Worldbuilder)
  - Schema: 力量体系/地理/文化/历史/种族
  - 关注: 设定规则、体系逻辑、世界观内部一致性
  - Prompt: 你是小说世界观分析专家...

角色2: 人物传记作者 (Character Analyst)
  - Schema: 角色名/别名/属性/关系网络/弧光/状态变化
  - 关注: 人物塑造、关系演变、角色成长轨迹
  - Prompt: 你是小说人物分析专家...

角色3: 叙事分析师 (Narrative Analyst)
  - Schema: 情节线/伏笔/转折点/主题/亮点/节奏
  - 关注: 叙事结构、节奏设计、核心吸引力
  - Prompt: 你是小说叙事分析专家...
```

**CREFT论文**的顺序多agent架构进一步验证：每个agent负责一个抽取维度（关系类型、关系强度、时间演化），顺序执行避免信息干扰。

[来源: arxiv.org/html/2604.01529v1] ✅ 论文自含实验

## 三、对话式多维关系抽取

arXiv:2507.04852：基于对话的小说角色关系抽取方法。

**核心洞察**：小说中角色关系主要通过对话塑造，传统纯叙述文本抽取会遗漏大量信息。从角色对话中提取多维关系：情感维度、权力维度、亲疏维度。

**对蒸馏方案的启示**：蒸馏过程应区分叙述文本和对话文本，对话文本使用专门的关系抽取prompt。

[来源: arxiv.org/abs/2507.04852]

## 四、MCP + RAG 架构方案

### 4.1 FastMCP + ChromaDB 架构

Suman Saha（2026年2月，TowardsAI）完整实践方案：

| 组件 | 技术选择 | 参数 |
|------|---------|------|
| MCP Server | FastMCP Python | 端口8081 |
| 向量数据库 | ChromaDB | 本地运行 |
| 嵌入模型 | nomic-embed-text | 本地运行 |
| 分块策略 | 500词/50词重叠 | 相关性阈值10% |
| MCP Client | Claude | 自动检索 |

Claude 通过 `project knowledge search tool` 自动检索相关内容。

[来源: towardsai.net] ⚠️ 单源但有完整代码

### 4.2 其他MCP方案

- **hyunjae-labs/lore**：Claude Code对话历史语义搜索MCP，嵌入模型索引过往对话
- **Lookio RAG MCP**：RAG知识库管理完整生命周期（上传/分块/索引/检索）
- **context-mode（用户已有）**：FTS5索引+子进程沙箱，适合关键词精确匹配
- **novel-db（用户已有）**：含 distillation_evolve、world_upsert、character_create 等全套小说蒸馏工具

### 4.3 双路检索建议

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| FTS5 (context-mode) | 精确关键词匹配 | 无语义理解 | 角色名、地名、技能名 |
| 向量搜索 (ChromaDB) | 语义相似度 | 需要嵌入模型 | "与火元素相关的能力" |
| **混合检索** | 兼顾精确+语义 | 复杂度高 | **生产环境最优解** |

## 六、推荐实施路径

### Phase 1 — 快速验证（1-2天）
- 百万字小说上传 Claude Projects
- 用内置 RAG 10倍扩展测试查询效果
- 验证"直接查原文"的可行性下限
- 成本：仅 API 调用费

### Phase 2 — 结构化蒸馏 MVP（3-5天）
- 基于 novel-db MCP + Grammar Builder + 角色化Agent
- 定义3个JSON schema（世界观/人物/情节）
- 逐章蒸馏，输出存入 novel-db 结构化表
- 成本：取决于章节量和模型选择

### Phase 3 — GraphRAG增强（可选，1-2周）
- 蒸馏结果导入 Neo4j 构建知识图谱
- LazyGraphRAG按需深入，避免全量索引成本
- 定制化entity-extraction prompt（按小说类型）

### Phase 4 — 混合检索
- 蒸馏知识图谱（精确结构查询）
- ChromaDB向量索引（语义搜索）
- 原文FTS5（关键词搜索）
- 三路并行，长上下文模型做最终综合
