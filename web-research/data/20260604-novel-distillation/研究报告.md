# 百万字网文小说蒸馏：深度调研报告

> 调研日期：2026-06-04 | 3 Agent × Sonnet | 20+ 信息源

## 背景与目标

需求：从百万字网文小说中蒸馏出世界观、设定、人物塑造、能力体系、核心亮点等结构化知识，支持后续精准搜索/引用/总结。本报告调研现有工具、技术方案和实现路径。

## 核心发现

### 1. 目前没有开箱即用的"小说蒸馏"工具

GitHub 搜索多次未找到高星项目。现有方案均为**通用 GraphRAG/RAG 工具 + 手动适配**。最接近的是微软 GraphRAG（15.9K stars）在中文网文上的实操案例：《仙逆》全本蒸馏（知乎博主 Mountain，DeepSeek 免费500万Token）和《凡人修仙传》知识图谱（B站播放19033）。

### 2. GraphRAG 是小说蒸馏的最成熟技术路线

微软 GraphRAG Pipeline：文本→LLM实体/关系提取→Leiden社区检测→递归摘要→向量嵌入。已有《仙逆》《凡人修仙传》《红楼梦》实操案例。但四大挑战：(a) 同一人物多别名 (b) 虚构地名NER识别差 (c) 人物关系动态变化 (d) 默认prompt未针对中文虚构文本优化。[来源: arXiv:2404.16130, github.com/microsoft/graphrag]

### 3. DOS RAG + MiA-RAG 是分块策略的2025新范式

斯坦福 DOS RAG（EMNLP 2025）：保持原文段落结构比切碎重组效果好2-8个点。MiA-RAG（中科院+腾讯）：三级分层摘要（全局→卷→章）使14B模型击败72B。结论：小说应**按章节分块保留原始结构**，同时构建**三级摘要支架**。[来源: arXiv:2506.03989, arXiv:2512.17220]

### 4. 1M token 上下文 ≠ 可靠检索

MRCR v2基准：Claude Opus 4.6@1M可靠率76%，GPT-5.4仅36.6%，Gemini 2.5 Pro仅16.4%。混合架构（RAG检索+长上下文综合）是生产环境默认方案。[来源: yage.ai, mindstudio.ai]

### 5. 医疗/法律领域的方法论可直接迁移

Nature 论文 LLM-AIx 的 **Grammar Builder**（JSON schema约束输出）+ arXiv:2604.01529的**角色化Agent分工**（效果提升2-3倍）。映射到小说："世界观架构师"+"人物传记作者"+"叙事分析师"三角色，各自带专属JSON schema。[来源: nature.com, arxiv.org]

### 6. 用户已有的 novel-db MCP + distillation_evolve 是最佳起点

用户自建 novel-db MCP 已具备蒸馏基础设施。结合 Grammar Builder + 角色化架构可增强：添加JSON schema约束、角色化prompt模板、结构化存储。

## 💡 意外发现

1. **LazyGraphRAG按需深入** — 先建实体图骨架，感兴趣部分按需深入，避免全量索引天文token成本
2. **Novelcrafter"场景后推荐Codex更新"可迁移** — 每处理完章节自动检测实体状态变化
3. **AI写作工具（Novelcrafter/Jenova/Dunia）均为文档式持久化，无知识图谱导出能力**
4. **Claude Projects内置RAG 10倍扩展是最快验证路径** — 上传百万字直接测试内置RAG效果

## ⚠️ 知识空白

- 小说专用分块策略缺乏实验数据（所有benchmark基于新闻/维基）
- 虚构文本实体抽取准确率无基准（precision/recall/F1）
- 动态实体状态追踪方案空白（人物属性随情节变化）
- 百万字蒸馏token成本无公开估算
- 小说蒸馏效果评估标准缺失（无"正确答案"可对比）

## 结论与建议

1. **快速验证（1-2天）**：百万字小说上传 Claude Projects，用内置 RAG 测试"直接查原文"的可行性下限
2. **结构化蒸馏 MVP（3-5天）**：基于 novel-db MCP + Grammar Builder + 角色化Agent，定义3个JSON schema（世界观/人物/情节），逐章蒸馏
3. **GraphRAG增强（可选，1-2周）**：蒸馏结果导入 Neo4j 构建知识图谱，LazyGraphRAG按需深入
4. **混合检索**：蒸馏知识图谱(精确结构查询) + ChromaDB向量(语义搜索) + 原文FTS5(关键词) 三路并行
