# 02-RAG分块策略与超长上下文方案

## 一、DOS RAG：保留原文结构碾压复杂Pipeline

斯坦福大学 Laitenberger, Manning, Liu 发表于 EMNLP 2025（arXiv:2506.03989）。

**核心发现**：保持文档原始段落顺序的简单 retrieve-then-read 方法，在 infinityBench 上比 ReadAgent、RAPTOR 等复杂多阶段 pipeline **高2-8个点**。

**关键洞察**：
- 复杂的文档摘要/层次化索引反而引入信息损失
- LLM 对原始文档结构的理解能力被低估
- 简单方法 + 原始结构保持 = 最优解

**对小说蒸馏的启示**：不应过度碎片化小说文本。按原始章节结构保留上下文，比切成小块再重组效果更好。按章节分块 + 500-1000词重叠是最合理的起点。

[来源: arxiv.org/html/2506.03989v2, github.com/alex-laitenberger/dos-rag-eval] ✅ 论文+代码双确认

## 二、MiA-RAG：三级"心智景观"摘要

中科院 + 腾讯微信AI（arXiv:2512.17220）。

**核心创新**：先对整篇文档建立分层摘要（全局→局部→段落三级），形成"心智景观"(mindscape)。检索时同时检索原文和对应层级的摘要。14B模型击败72B基线。

**映射到小说蒸馏**：

```
全局摘要层 → 世界观大纲、核心主题、力量体系总览
  ↓
卷摘要层 → 大事件线、主要人物弧光、关键转折点
  ↓
章摘要层 → 情节细节、场景描写、人物互动
  ↓
原文层 → 原始章节文本（DOS RAG保留）
```

**与DOS RAG的互补关系**：DOS RAG说"不要切碎原文"，MiA-RAG说"需要分层摘要"。结合方案：保留原始章节文本 + 构建三级摘要支架，检索时同时命中原文和摘要。

[来源: arxiv.org/abs/2512.17220]

## 三、分块策略实证数据

基于多个RAG benchmark的综合分析：

| 策略 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| 固定大小分块 | 实现简单、可预测 | 可能切断语义单元 | 事实查询 |
| 语义分块 | 保持语义完整 | 仅比固定提升3-5% | 长文档理解 |
| 结构感知分块 | 利用文档结构 | 对结构化文档+15% | 有标题/章节文档 |
| 叙事大块 | 保留情节连贯性 | 单块信息量大 | **小说/虚构文本** |

**关键发现**：
- 分块大小的影响远大于分块方法（512→1200 tokens 比 semantic vs fixed 差异更大）
- 叙事文本宜用大块（保留情节连贯性），事实查询宜用小块
- 小说建议：**按章节分块 + 500-1000词重叠**

[来源: weaviate.io/blog/chunking-strategies-for-rag, 多篇benchmark]

## 四、Agentic RAG：2026第四代范式

综述论文 arXiv:2501.09136 定义的RAG演进路径：

```
Naive RAG (2023) → Advanced RAG (2024) → Modular RAG (2024) → Agentic RAG (2025-2026)
```

**Agentic RAG 特征**：
- 自主agent具备规划(planning)、反思(reflection)、工具使用能力
- 多agent协作：不同agent负责不同检索/分析维度
- 可动态选择检索策略

**映射到小说蒸馏**：
```
"世界观提取agent" — 负责力量体系、地理、文化设定
"人物关系agent" — 负责角色属性、关系网络、人物弧光
"情节分析agent" — 负责事件线、伏笔、叙事结构
```

[来源: arxiv.org/html/2501.09136v4]

## 五、超长上下文模型实测对比

### 5.1 1M token 可靠检索率（MRCR v2基准）

| 模型 | 上下文窗口 | 1M可靠检索率 |
|------|-----------|-------------|
| Claude Opus 4.6 | 1M tokens | **76%** |
| GPT-5.4 | 1M tokens | 36.6% |
| Gemini 2.5 Pro | 1M tokens | 16.4% |

**核心结论**：上下文窗口大小与实际可靠检索能力是两回事。Gemini虽然"支持"1M但可靠性远低于Claude。

[来源: yage.ai/share/long-context-benchmark-en-20260315.html]

### 5.2 2026年超长上下文模型全景

| 模型 | 窗口 | 有效容量比 | 1M处理成本 | 延迟 |
|------|------|----------|-----------|------|
| Llama 4 Scout | 10M | ~60% | 开源免费 | 本地 |
| Gemini 3.1 Pro | 1M | ~65% | $2.00 | 中等 |
| Claude Opus 4.6 | 1M | ~70% | $5.00 | 较慢 |
| Qwen2.5-1M | 1M | ~60% | 开源免费 | 本地 |

**有效容量**仅为广告值的60-70%。8万字以上Gemini几乎唯一选择（100万token窗口一次性消化整份文档）。分层提问比一次性提问好：大纲→具体章节→跨章节关联。

[来源: digitalapplied.com, cloud.tencent.com/developer/article/2673693]

### 5.3 1M上下文 vs RAG 对比

MindStudio实测结论：
- 1M窗口适合"偶尔查一次大文档"场景
- 成本：Gemini $0.15/1M input tokens/call，但延迟30s+
- RAG在频繁查询场景下成本/延迟碾压纯长上下文
- **混合架构成为生产默认**：RAG检索(高效低成本) → 长上下文模型综合(推理能力)

[来源: mindstudio.ai]

## 六、Lost in the Middle 问题

Stanford/UW 2023基础论文：LLM对长文本中间位置信息检索可靠性显著下降。模型对开头和结尾信息检索准确，但中间位置信息容易被"遗忘"。

**对百万字小说的影响**：关键设定/人物介绍如果出现在小说中段，RAG检索时可能遗漏。

**缓解方案**：
- MiA-RAG的分层摘要确保全局信息不被遗漏
- DOS RAG的原始顺序保留维持上下文连贯
- 分章节独立索引避免单个chunk过长

## 七、Claude Projects 内置 RAG

Claude官方文档（2026年3月）：project knowledge超出上下文窗口时RAG自动激活，扩展容量最高10倍。使用专用的"project knowledge search tool"进行检索。

**对小说蒸馏的价值**：
- ✅ 最快验证路径：直接上传百万字小说测试查询效果
- ✅ 零配置自动检索
- ❌ 无法精细控制分块策略
- ❌ 无法做结构化抽取（只能检索原始文本）
- ❌ 知识无法持久化为独立知识图谱
- 结论：RAG查原文 + 蒸馏建知识库，两者互补
