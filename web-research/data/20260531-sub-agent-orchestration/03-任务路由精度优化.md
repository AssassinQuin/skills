# 主题3: 子Agent任务路由精度优化（学术研究+实践方案）

> 调研日期: 2026-05-31 | 来源数: 7 | 交叉验证: ✅ 5源确认

## 核心发现

### 3.1 路由精度现状：80-82%是学术SOTA

**结论：当前学术最佳路由准确率约80-82%（Critique-and-Router, AdaptOrch），生产环境75%+可通过良好prompt工程实现。**

| 方法 | 准确率 | 来源 | 验证方式 |
|------|--------|------|---------|
| Critique-and-Router (GRPO训练) | 80.5-82.3% | [arXiv 2605.08686] | 两个测试集 |
| AdaptOrch拓扑路由 | 81.2% | [arXiv 2602.16873] | SWE-Bench |
| 良好prompt+few-shot（无训练） | 75%+ | [Critique-and-Router论文] | 理论推导 |
| 成熟路由系统（结构化任务） | 85-95% | [DEV Community] | 行业报告 |

### 3.2 AOrchestra: 4元组Phi=<Instruction, Context, Tools, Model>

**核心贡献**：框架无关的动态子Agent创建抽象，将路由问题分解为4个维度的联合优化。

| 维度 | 对books_creater的映射 | 当前实现 |
|------|---------------------|---------|
| Instruction | skill触发词+任务描述 | ✅ skill系统已有 |
| Context | 章节上下文包 | ✅ get_chapter_context |
| Tools | MCP工具集 | ✅ 工具白名单 |
| Model | haiku/sonnet/opus分配 | ❌ Agent工具不支持model参数 |

**Benchmark数据**：
- GAIA: 80% pass@1（比最强基线+16.28%）
- SWE-Bench: 82%
- SFT提升编排质量: +11.51%

💡关键发现：SFT只需要"少量expert轨迹"就能大幅提升。意味着收集5-10个高质量写作编排案例就能训练出更好的路由器。

[✅ arXiv 2602.03786, 2025-02]

### 3.3 AdaptOrch: 拓扑优化 > 模型选择（高精度场景）

**Performance Convergence Scaling Law**：

```
Var_τ / Var_M >= Ω(1/ε²)
```

当精度要求ε→0时，编排拓扑优化的方差主导模型选择的方差。

**对books_creater的启示**：
- 追求小说写作高精度时，**优化skill编排拓扑比升级到Opus更重要**
- 当前6个skill的调用拓扑（用户选择→skill执行→MCP调用）已经是简化版层次拓扑
- 可以进一步优化：哪些skill可以并行、哪些必须串行、检查点在哪

**关键数据**：
- 拓扑路由开销 <50ms（远低于LLM推理2-15s）
- SWE-Bench改进 22.9%
- 4种拓扑：Parallel/Sequential/Hierarchical/Hybrid

[✅ arXiv 2602.16873, 2025-02]

### 3.4 非对称损失函数：下路由的代价远大于上路由

| 路由错误 | 后果 | 可恢复性 | 经济影响 |
|---------|------|---------|---------|
| 上路由（简单任务→强模型） | 浪费金钱（10-50x成本差距） | 可恢复 | 浪费但安全 |
| 下路由（复杂任务→弱模型） | 质量下降 | 不可恢复 | 可能致命 |

**实践建议**：路由器应偏向保守——宁可让haiku做sonnet的活（浪费一点），不可让haiku做opus的活（质量崩盘）。这与coding-rules R5.1的映射一致。

[✅ AOrchestra + AdaptOrch综合分析]

### 3.5 Anthropic官方路由建议

1. **路由分类器用简单模型**：路由决策不需要复杂推理，haiku级别足够
2. **Evaluator-Optimizer循环**：一个Agent生成，另一个评估，循环直到达标。端到端精度可远超路由器精度
3. **不要过度复杂化**："简单好提示90%准确率时，Agent系统只需处理剩余10%"
4. **Compounding Error风险**：过度工程化的Agent系统可能引入更多累积误差

[✅ Anthropic "Building Effective Agents" + "Effective Context Engineering"]

### 3.6 Context Engineering对路由精度的间接影响

| 机制 | 对路由的影响 | 实践方案 |
|------|------------|---------|
| Context Rot（上下文腐化） | 长对话后路由判断质量下降 | 结构化笔记对抗 |
| Attention Budget | 中间位置注意力最低 | subagent独立窗口规避 |
| Sub-agent架构 | 每个子Agent精简上下文 | 提升单任务精度 |

**Anthropic观察**：使用sub-agent架构后，单Agent在长任务上的pass rate显著提升。

[✅ Anthropic "Effective Context Engineering"]

## 💡灵感

1. **Few-shot路由prompt**：Critique-and-Router证明，在路由prompt中加入few-shot示例可达75%+准确率。可在Agent工具的prompt参数中直接实现——不需要修改代码，只需要写好prompt模板。这是ROI最高的优化路径。
2. **编排轨迹收集**：AOrchestra的SFT发现（+11.51%）意味着我们的novel-write skill可以通过记录每次写作的skill调用序列+质量评分，逐步积累编排expert轨迹。这比换模型更能提升长期质量。
3. **保守路由策略**：非对称损失函数告诉我们，在不确定时应该分配更强的模型。当前R5.1的haiku/sonnet/opus映射已经体现了这个原则（不确定时用sonnet而非haiku）。
