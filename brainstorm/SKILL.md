# 头脑风暴引擎

<what-to-do>

## 强制流程

```
Phase 0: 需求深挖 → 反驳/追问/逼出真问题 → 🔒需求确认
Phase 1: 上下文采集 → 读取已有内容 → 搜索调研(sonnet agent) → 🔒搜索确认
Phase 2: 方法选择 → 问题分类 → 选方法+验证有效性 → 🔒方法确认
Phase 3: 深度发散 → spawn sonnet agent逐轮发散 → 主对话收摘要+展示 → 💡灵感捕捉
Phase 4: 综合收敛 → 保存成果(data/目录+memory) → 🔒确认输出
```

</what-to-do>

<supporting-info>

## Phase 0: 需求深挖（不可跳过）

**核心原则：用户说的大概率不是真需求。** 不追问就开脑洞 = 浪费所有人时间。

### 执行步骤

1. **听完用户需求后，先质疑**：
   - "你说的X，到底想解决什么？X可能只是表面症状"
   - "为什么需要这个？背后的动机是什么？"
   - "你试过什么？为什么不行？卡在哪？"

2. **逼出3个关键信息**（缺任何一个都要追问）：
   - **真问题**（不是表面需求，是深层动机）
   - **已有方案/限制**（现在有什么？不能动什么？）
   - **成功标准**（什么叫"好了"？怎么衡量？）

3. **犀利但有用**：不是抬杠，是帮用户想清楚。每条追问都带着一个假设或方向。

4. **如果用户给的信息足够清晰**（有具体背景+有约束+有目标）→ 直接确认，不多问。

### 🔒 需求确认

```
【需求确认】
你说的是：{用户的原话}
我理解为：{真问题}
约束条件：{已有方案/限制}
成功标准：{什么叫好了}
确认 / 补充 / 纠正？
```

### 关键事实清单（必填，在 🔒 需求确认 之前或同时输出）

**目的：防止整个 brainstorm 建立在错误的前提上。**

根据任务领域，列出必须确认的**具体事实**（不是抽象约束）：

```
【关键事实确认】
1. {事实1：具体的关系/时间线/机制/数值}
2. {事实2}
3. {事实3}
...
确认 / 补充 / 纠正？
```

**小说领域（必填）**：
- 涉及的所有角色 + 角色之间的关系（谁是盟友/敌人/中立？谁和谁一伙？）
- 关键时间线（事件先后顺序，谁先出现谁后出现）
- 不可改的具体设定（不是"约束条件"这种抽象词，是"封绝是裔族""封绝+虞渊对抗渊"这种硬事实）

**技术/产品领域**：
- 当前状态（现有方案的技术栈、架构、数据流）
- 目标状态（期望达到什么效果，有具体指标吗？）
- 不可动的部分（哪些必须保留？）

**原则**：宁可多列一条被用户说"不用确认"，也不要漏掉一条导致整个方向跑偏。

### 用户纠正机制（贯穿全流程）

**如果用户纠正的是方向/偏好/风格** → 在当前方案上调整，继续推进。

**如果用户纠正的是事实/关系/前提** → 必须执行：
1. 立即停止当前阶段
2. 回到关键事实清单，重新确认所有受影响的事实
3. 标记已产出的创意为"前提有误，需重检"
4. **禁止在旧方案上做最小修改**——必须从确认过的事实重新推导

**判断标准**：如果用户的纠正让已有创意的逻辑链断裂，就是事实级纠正。

### 小说领域增强

当检测到小说/角色/世界观/剧情/力量体系/反派等信号时，立即加载 [references/novel-brainstorm.md](references/novel-brainstorm.md)。追问时额外问：
- "你现在的设定是什么？（给我文件路径或概要）"
- "涉及哪些角色？他们之间是什么关系？"
- "关键时间线是什么？先后顺序？"
- "你想让读者感受到什么？"
- "什么是绝对不能改的？"（要具体到事实，不要抽象约束）

## Phase 1: 上下文采集

### 读取已有内容

读取用户提到的文件/项目，提取关键信息作为头脑风暴种子。

### 搜索调研（需要时）

**判断是否需要搜索**：
- 涉及特定领域知识 → 需要
- 涉及竞品/参考案例 → 需要
- 用户有足够上下文 → 不需要
- 用户说"不用搜" → 跳过

**搜索时用低模型降低成本**：
- 搜索+提取 → spawn sonnet agent（`model: "sonnet"`）
- 简单事实查询 → 直接用当前模型搜索（不值得开agent）

🔒 展示搜索计划，确认后执行：
```
【搜索计划】
方向1: {query} — {为什么需要}
方向2: {query} — {为什么需要}
模型: sonnet agent
预计: {N}次调用
确认？
```

## Phase 2: 方法选择

### 问题分类决策树

```
问题类型？
├── 改进现有方案/设定 → SCAMPER（7动词改造）
├── 遇到矛盾/两难 → TRIZ（矛盾消除）
├── 需要全新方向 → 侧向思维（PO/随机输入/类比）
├── 多维度系统性探索 → 形态分析（矩阵组合）
├── 想不出来/卡住了 → 自适应解卡（见Phase 3）
├── 需要推翻既有假设 → 第一性原理（分解→去假设→重构）
├── 需要全面评估 → 六顶思考帽（6角色审视）
├── 从结果倒推 → 逆向思维（反向推导）
└── 跨领域寻找灵感 → 仿生学/概念融合
```

每次至少选 **2种互补方法**：1种发散 + 1种评估/变换视角。

### 方法有效性验证（防止无效指令）

选方法后，**自我验证**：
1. 这个方法能直接产出用户要的东西吗？（如果不能 → 换方法）
2. 我有足够信息执行这个方法吗？（如果没有 → 回Phase 1补信息）
3. 这个方法会不会产出一堆废话？（如果会 → 加约束条件）

**方法详情**：[references/methods.md](references/methods.md)，按需加载。

### 🔒 方法确认

```
【头脑风暴计划】
核心问题：{一句话}
问题类型：{分类}
选择方法：
  Round 1: {方法A} — {为什么选这个} — {预期能产出什么}
  Round 2: {方法B} — {为什么选这个} — {预期能产出什么}
有效性验证：{为什么这两方法不是在浪费时间}
确认？
```

## Phase 3: 深度发散

### 子 agent 执行（避免上下文爆炸）

每轮发散 spawn 一个 sonnet agent 执行，主对话只收摘要：

```
Agent prompt 结构：
  角色：创意发散专家
  任务：用{方法名}对"{核心问题}"进行深度发散

  前提事实（如果以下任何一条有误，你的所有推论都无效）：
  1. {从关键事实清单中提取的、与本次发散直接相关的事实}
  2. {事实2}
  ...
  在你开始发散前，确认你理解了这些事实。如果不确定某条事实的含义，在输出开头标注疑问。

  上下文：{Phase 0-1 收集的需求+背景+约束}
  输出要求：
    - 至少 5 个不同方向的创意（每个必须是具体可描述的方案）
    - 每个创意必须说明具体机制（不能只给标签/修辞，要回答"怎么做到的"）
    - 每个创意标注：方向标签 + 一句话说明 + 可行性/创新性/惊喜度(1-5)
    - 标记💡灵感发现（反常识/跨领域/有趣/追问）
    - 如果创意都在安全区，强制用"如果相反"反转一次
  约束：{用户约束条件}
```

主对话收到 agent 结果后：
1. 展示摘要给用户（Top 创意 + 💡灵感）
2. 如果饱和度不足（<3个不同方向）→ 换方法再开一轮 agent
3. 收集所有轮次结果进入 Phase 4

### 发散轮次管理

- 每轮一个方法，每轮 spawn 一个 sonnet agent
- 最少 2 轮（2种方法），最多 4 轮
- 如果用户说"继续"或"再来一轮" → 追加轮次

### 自适应解卡（注入到 agent prompt）

| 状态 | 注入策略 |
|------|---------|
| 卡壳 | "随机从{无关领域}取一个概念强制嫁接" |
| 太保守 | "删掉最大约束，在无约束条件下重新发散" |
| 太发散 | "回到用户原始需求{需求}，只保留可落地的创意" |

### 灵感标记

- 🔍 **反常识** — 与直觉矛盾的好东西
- 🔗 **跨领域** — 其他领域的意外关联
- ⚡ **有趣** — 让人眼前一亮
- ❓ **追问** — 引出新问题

### 自适应解卡策略

| 状态 | 信号 | 策略 |
|------|------|------|
| 卡壳 | 创意重复/质量下降 | 随机输入 + 换完全不同的领域类比 + 换POV |
| 太保守 | 都在安全区内 | 假设反转 + "如果相反" + 删掉最大约束 |
| 太发散 | 无法落地 | 六帽黑帽收敛 + 加约束 + 回到用户原始需求 |
| 需要搜索 | 创意需要领域知识 | spawn sonnet agent搜索（当前模型继续发散） |

### 灵感标记

- 🔍 **反常识** — 与直觉矛盾的好东西
- 🔗 **跨领域** — 其他领域的意外关联
- ⚡ **有趣** — 让人眼前一亮
- ❓ **追问** — 引出新问题

## Phase 4: 综合收敛 + 持久化

1. 去重合并相似创意
2. 按 **可行性/创新性/惊喜度** 三维度评分
3. 标注灵感亮点
4. **反废话过滤**：每条成果必须能回答"具体机制是什么？"。标签和修辞不能作为成果正文，只能作为概括。如果回答不了"怎么做到的"→ 删除或补充具体机制
5. **事实回检**：用关键事实清单验证最终成果中的每条关键逻辑，标记与已确认事实矛盾的创意
6. 🔒 展示最终成果

### 持久化（每条成果必须执行）

**1. 保存完整成果到 skill 目录**：

```
~/.claude/skills/brainstorm/data/{YYYYMMDD}-{slug}/
├── 成果.md          # 完整创意清单+历程+灵感
└── raw/
    ├── round-1.md   # 各轮 agent 原始输出
    └── round-2.md
```

- slug：从核心问题提取，如 `反派组织设计` → `villain-org-design`
- 成果.md 使用下方输出格式
- raw/ 下保存各轮子 agent 的完整输出

**2. Memory 概要保存**：

```
memory_store(
  content: "头脑风暴[{主题}]: {核心问题}。Top创意: 1){最佳创意1} 2){最佳创意2} 3){最佳创意3}。💡{最有趣的灵感}。详情: ~/.claude/skills/brainstorm/data/{路径}",
  metadata: { tags: "brainstorm,creative", type: "brainstorm-result" }
)
```

**3. 项目级记忆**（如在项目目录下运行）：
- 同样用 memory_store，额外加项目 tag
- 小说场景：结果可同步到项目灵感库目录

### 输出格式

```
# 头脑风暴成果：{主题}

{一句话}

- {意外发现}

## 历程回顾
- Round 1 ({方法}): {产出摘要} → {转折/发现}
- Round 2 ({方法}): {产出摘要} → {转折/发现}
- 💡 最有价值的转折：{什么}

## 下一步
- {可探索方向}
```
